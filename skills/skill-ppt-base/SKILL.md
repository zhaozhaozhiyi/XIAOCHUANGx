# PPT · 需求采集基座

## 目标

在进入具体 PPT 模版或生成引擎之前，先把用户的自然语言需求收敛成**可执行的演示 brief**，再进入页纲与交付物生成。

这个 Skill 负责**流程**，不负责最终视觉风格。风格、结构偏好、具体产出规范由后续选中的 PPT 模版 Skill 承担。

## 何时使用

- 新建 PPT 会话时，作为默认基座流程
- 用户只给出一句笼统需求，信息不足以直接生成 PPT
- 用户提供了文稿或材料，但仍缺少受众、目标、页数、风格等关键信息
- 需要先做需求摘要确认，再进入页纲阶段
- 用户已经指定具体 PPT 模板或 PPT Skill 时，仍先由本基座 Skill 负责流程判断与交接

## 核心原则

1. **只问缺失信息**：不要把用户已经明确提供过的信息再问一遍。
2. **少而关键**：每轮最多追问 1～4 个高杠杆问题，优先补齐主题、场景、受众、目标。
3. **AI to UI 优先**：若运行时支持结构化问题卡，优先返回结构化问题给前端渲染；否则退化为简短编号追问。
4. **先确认再生成**：首轮信息充分时，先输出需求摘要等待用户确认；若用户是在需求表单/追问之后补齐信息，则该补充可视为 brief 输入已确认，可输出摘要并继续进入页纲与交付物生成。
5. **模版 Skill 管风格**：本 Skill 完成 brief 收敛后，只选择并交接一个主模版 Skill；不要混用多套风格。
6. **信息充分则不追问**：若用户首轮已经提供足够进入 PPT brief 的核心信息，禁止再输出 `ppt_requirements` 卡片或重复追问，必须直接进入“需求摘要确认”。
7. **指定模板不跳过基座**：用户指定模板 / Skill 时，仍由本 Skill 先判断信息是否充分、是否需要摘要或页纲，再把任务交给目标 PPT Skill；不允许前端直接绕过基座 Skill。
8. **信息充分时轻通过**：若用户已指定模板 / Skill 且信息已经充分，本 Skill 应尽量轻量通过，不强行发表单或追加无必要追问。
9. **控制 token 重复消耗**：本 Skill 的职责是收敛需求、形成 brief、决定是否进入页纲，再交给下游 Skill；不要与下游 Skill 重复做完整需求理解、重复追问或重复长篇复述。

## 必经流程

### 1. 判断信息是否充分

先判断当前是否已经具备进入 PPT 生产的最低信息：

- 演示主题
- 使用场景
- 目标受众
- 汇报目标

若以上四项都已基本明确，可直接进入“需求摘要确认”；缺失时再追问。

判定为“已基本明确”后：

- 不再补发需求表单
- 不再把已知信息改写成追问
- 直接整理为需求摘要，等待用户确认

### 2. 结构化追问

若信息不足：

- 优先按 [references/questionnaire-protocol.md](./references/questionnaire-protocol.md) 返回结构化问题
- 问题数量控制在 1～4 个
- 问题类型仅使用协议中定义的稳定类型
- 一轮追问后仍有少量缺失时，可基于合理默认值补足，并在摘要中明确标注“默认假设”

若运行时还不支持结构化问题卡，则退化为：

```markdown
为了把 PPT 做得更准，我还需要补 3 个信息：
1. 这次是给谁汇报？
2. 更偏方案提案、研究汇报，还是阶段复盘？
3. 预期控制在多少页以内？
```

### 3. 生成需求摘要并等待确认

收到用户补充后，整理成一份**需求摘要**，格式尽量稳定：

```markdown
## PPT 需求摘要
- 主题：
- 场景：
- 受众：
- 目标：
- 页数：
- 风格：
- 参考材料：
- 必须保留内容：

请确认以上理解是否正确。我确认后再进入页纲。
```

若这是首轮信息充分后直接生成的摘要，等待用户确认后再进入下一步。

若这是用户对 `ppt_requirements` 表单或编号追问的补充答案，则把这次补充视为“已确认 brief”：

- 必须仍输出 `ppt_requirement_summary`，让前端沉淀摘要卡
- 不要再次要求用户确认同一批信息
- 继续输出 `ppt_outline`，让前端渲染可编辑页纲
- 按用户已补齐的 brief 进入交付物生成，至少产出 HTML 预览；可行时同步产出 PPTX

### 4. 选择主模版 Skill

需求确认后，按 [references/template-routing.md](./references/template-routing.md) 选择一个主模版 Skill：

- 路演 / 融资 / 客户提案：优先 `skill-ppt-pitch-deck`
- 周报 / 阶段状态同步：优先 `skill-ppt-weekly-report`
- 季度复盘：优先 `skill-ppt-quarterly-review`
- 技术分享：优先 `skill-ppt-tech-sharing`
- 框架梳理 / 知识结构：优先 `skill-ppt-knowledge-arch`
- 无明显场景：退回 `skill-ppt-deck`

只选择一个主模版 Skill，并把确认后的需求摘要作为后续输入基座。

若用户已经明确指定目标模板 / Skill：

- 优先尊重用户指定
- 本 Skill 仍负责完成需求收敛与摘要确认
- 若信息已足够，则直接轻量交接给指定 Skill
- 不因“已选模板”而跳过需求判断，也不因“走基座”而强行增加追问
- 交接给下游 Skill 时，优先传递结构化 brief，而不是把整段历史原样重复展开

## Token 使用原则

- 本 Skill 应被视为轻量流程编排层，不是第二个完整 PPT 助手
- 信息充分时，使用最短必要输出完成摘要确认或直接交接
- 下游 PPT Skill 应消费已确认的 brief，避免重新收集同一批需求
- 除非确有必要，不要把首轮输入、追问文本、摘要文本、页纲说明全文重复拼接给下游

### 5. 输出页纲并等待确认

在交接给模版 Skill 前或交接时，先产出页纲：

- 推荐封面标题
- 建议页数
- 每页标题
- 每页 3～5 个核心点
- 哪些页需要图表、数据、案例或结论页

首轮直接生成的页纲要等待用户确认或修订后，才能进入最终生成。若页纲来自用户已提交需求表单后的连续流程，可先输出可编辑 `ppt_outline`，同时继续按该页纲生成初版交付物，后续允许用户再改。

若运行时支持 `parts[]` 结构化渲染，则输出页纲时必须把**完整页纲正文**包在以下隐藏标记中，方便 Companion 解析为 `ppt_outline` 卡片：

```markdown
<!--JLC:PPT_OUTLINE_START-->
# 封面标题建议

1. 第 1 页：封面
2. 第 2 页：背景
3. 第 3 页：核心判断
<!--JLC:PPT_OUTLINE_END-->
```

要求：

- 标记内只放页纲正文，不要放额外解释
- 标记外可以补一句很短的引导语，例如“下面是建议页纲”
- 若未进入页纲阶段，不要输出该标记

## 需求充分性的判断标准

以下条件满足时，可认为足够进入页纲阶段：

- 用户要讲什么：明确
- 讲给谁听：明确
- 这套 PPT 想达成什么：明确
- 不是完全没有材料支撑，或允许你按常规结构先出框架：明确

页数、风格、参考材料如果缺失，可以：

- 使用默认值
- 或在摘要中显式写出假设

但主题、场景、受众、目标这四项尽量不要缺。

若这四项在首轮输入中已经齐备，即使页数、风格、材料尚未完全明确，也应优先：

- 基于已知信息生成摘要
- 对次级缺失项使用默认假设或在摘要中标注待确认
- 不再退回到 `ppt_requirements` 表单

### 首轮固定回归样例

以下样例用于约束首轮判断行为。遇到同类输入时，必须按期望阶段推进：

| 编号 | 输入特征 | 期望行为 |
|------|----------|----------|
| F4 | `做一份面向客户高层的金联创产品能力介绍 PPT，用于售前沟通，目标是说明数据、研究和交付优势，12 页左右。` | 信息充分，不输出 `ppt_requirements`；直接生成 `ppt_requirement_summary` |
| F5 | `帮我做个公司介绍 PPT。` | 信息不足，输出 `ppt_requirements`；只追问场景、受众、目标、页数或风格等缺失项 |
| F6 | `基于刚才的文稿生成一份 10 页汇报 PPT，面向内部经营会，重点突出市场判断和行动建议。` | 信息充分，不输出 `ppt_requirements`；进入摘要或页纲阶段 |

### 连续流固定回归样例

以下样例用于约束“需求卡提交后继续执行”的行为：

| 编号 | 输入与补充 | 期望行为 |
|------|------------|----------|
| T5 | 用户先说 `帮我做个公司介绍 PPT。`，随后补充 `场景：售前沟通；受众：客户高层；目标：介绍公司能力并推动下一步交流；页数：12 页；风格：专业商务` | 输出 `ppt_requirement_summary`，再输出带隐藏标记的 `ppt_outline`，并生成 `.html` 预览；可行时同步生成 `.pptx` |

硬性约束：

- 只要主题、场景、受众、目标已经能从首轮输入中合理识别，就不要再发需求卡
- 页数、风格、参考材料缺失时，优先写入摘要中的默认假设，不作为必然追问理由
- 追问只能补缺失项，不得把已知的主题、场景、受众、目标改写成问题再问一遍
- 若输出结构化卡片，`part.kind` 必须是 `ppt_requirements`；若输出摘要，`part.kind` 必须是 `ppt_requirement_summary`

## 问题设计要求

- 只问会显著改变结构或内容的问题
- 问题文案要短，不要长段解释
- 单轮不要混入太多开放题
- 可选择的就不要都做成自由文本
- 若用户已上传材料，优先围绕材料用途提问，而不是重新问主题

## 输出要求

- 追问阶段：输出结构化问题或简短编号问题
- 摘要阶段：输出稳定的需求摘要；若支持结构化渲染，必须使用 `<!--JLC:PPT_REQUIREMENT_SUMMARY_START-->` / `<!--JLC:PPT_REQUIREMENT_SUMMARY_END-->` 包裹完整摘要正文
- 页纲阶段：输出可确认的 slide outline；若支持结构化渲染，必须使用 `<!--JLC:PPT_OUTLINE_START-->` / `<!--JLC:PPT_OUTLINE_END-->` 包裹页纲正文
- 生成阶段：交给选中的 PPT 模版 Skill 执行，并要求最终产物写入工作区；至少生成 `.html` 预览，可行时同步生成 `.pptx`
- 文件交付必须真实落盘：只有在实际创建或更新了工作区文件后，才能在最终回复中说“已生成/已完成”并给出路径；如果写入失败，必须明确说明失败原因，禁止声称已生成文件

## 禁止

- 不要在首轮就抛出 7～8 个开放问题
- 不要在首轮信息已经充分时仍输出 `ppt_requirements`
- 不要因为用户指定模板 / Skill 就让前端直接绕过基座流程
- 不要因为用户指定模板 / Skill 就机械补发表单
- 不要把基座 Skill 做成与下游 Skill 重复耗 token 的第二套完整 PPT 流程
- 不要在首轮需求未确认时直接生成完整 deck；用户提交需求表单后的连续流程除外
- 不要同时加载多套互相冲突的 PPT 风格 Skill
- 不要虚构“用户已提供”的材料、页数或风格偏好
- 不要虚构已经写入工作区的 `.html` 或 `.pptx` 文件路径
- 不要把追问写成纯闲聊，要让前端可稳定渲染或退化展示
