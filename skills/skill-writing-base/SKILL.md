# 写作 · 需求采集基座

## 目标

在进入具体写作 Skill 之前，先把用户的自然语言需求收敛成**可执行的写作 brief**，再按任务复杂度决定是直接成稿，还是先进入大纲后再成稿。

这个 Skill 负责**流程**，不负责最终文体细节。文体、结构、格式和成稿规范由后续选中的具体写作 Skill 承担。

## 何时使用

- 新建写作会话时，作为默认基座流程
- 用户只给出一句笼统需求，信息不足以直接开始成稿
- 用户提供了材料，但仍缺少体裁、受众、用途、篇幅等关键信息
- 需要先做需求摘要确认，再按需进入大纲阶段
- 用户已经指定具体写作模板或写作 Skill 时，仍先由本基座 Skill 负责流程判断与交接

## 核心原则

1. **只问缺失信息**：不要重复问用户已经说清楚的内容。
2. **少而关键**：每轮最多追问 1～4 个高杠杆问题，优先补齐主题、体裁、受众、用途。
3. **AI to UI 优先**：若运行时支持结构化问题卡，优先返回结构化问题给前端渲染；否则退化为简短编号追问。
4. **先确认再起稿**：首轮信息充分时，先输出需求摘要等待用户确认；若用户是在需求表单/追问之后补齐信息，则该补充可视为 brief 输入已确认，可输出摘要并继续进入大纲或成稿。
5. **成稿 Skill 管文体**：本 Skill 完成 brief 收敛后，只选择并交接一个主写作 Skill；不要在一轮中混用多套体裁规范。
6. **信息充分则不追问**：若用户首轮已经提供足够进入写作 brief 的核心信息，禁止再输出 `writing_requirements` 卡片或重复追问，必须直接进入“需求摘要确认”。
7. **指定模板不跳过基座**：用户指定模板 / Skill 时，仍由本 Skill 先判断信息是否充分、是否需要摘要或大纲，再把任务交给目标写作 Skill；不允许前端直接绕过基座 Skill。
8. **信息充分时轻通过**：若用户已指定模板 / Skill 且信息已经充分，本 Skill 应尽量轻量通过，不强行发表单或追加无必要追问。
9. **控制 token 重复消耗**：本 Skill 的职责是收敛需求、形成 brief、决定是否进入大纲，再交给下游 Skill；不要与下游 Skill 重复做完整需求理解、重复追问或重复长篇复述。

## 必经流程

### 1. 判断信息是否充分

先判断当前是否具备进入写作生产的最低信息：

- 文稿主题
- 文稿类型或体裁
- 目标受众
- 目标用途

若以上四项都已基本明确，可直接进入“需求摘要确认”；缺失时再追问。

判定为“已基本明确”后：

- 不再补发需求表单
- 不再把已知信息改写成追问
- 直接整理为需求摘要，等待用户确认

### 2. 结构化追问

若信息不足：

- 优先按 [references/questionnaire-protocol.md](./references/questionnaire-protocol.md) 返回结构化问题
- 问题数量控制在 1～4 个
- 问题类型仅使用协议中定义的稳定类型
- 一轮追问后仍有少量缺失时，可使用合理默认值，并在摘要中明确标注“默认假设”

若运行时还不支持结构化问题卡，则退化为：

```markdown
为了把这篇文稿写得更准，我还需要补 3 个信息：
1. 这篇文稿更偏研究报告、政策解读，还是正式公文？
2. 主要是写给谁看？
3. 希望最后控制在什么篇幅？
```

### 3. 生成需求摘要并等待确认

收到用户补充后，整理成一份**写作需求摘要**，格式尽量稳定：

```markdown
## 写作需求摘要
- 主题：
- 体裁：
- 受众：
- 用途：
- 篇幅：
- 风格：
- 参考材料：
- 必须保留内容：

请确认以上理解是否正确。我确认后再决定是否先出大纲，或直接进入起草。
```

若这是首轮信息充分后直接生成的摘要，等待用户确认后再进入下一步。

若这是用户对 `writing_requirements` 表单或编号追问的补充答案，则把这次补充视为“已确认 brief”：

- 必须仍输出 `writing_requirement_summary`，让前端沉淀摘要卡
- 不要再次要求用户确认同一批信息
- 简单短文可直接进入成稿并写入工作区
- 研究报告、长篇专题等复杂任务继续输出 `writing_outline`，并按用户已补齐的 brief 进入后续生成

### 4. 选择主写作 Skill

需求确认后，按 [references/skill-routing.md](./references/skill-routing.md) 选择一个主写作 Skill：

- 一般研究、分析、综合稿：优先 `skill-writing-general`
- 通知、请示、报告、函、意见等：优先 `skill-writing-official-doc`
- 会议纪要体裁：优先 `skill-writing-meeting-minutes`
- 政策解读：优先 `skill-wr-policy`
- 专题研究：优先 `skill-wr-topic`
- 行业研究：优先 `skill-wr-industry`
- 宏观数据解读：优先 `skill-wr-macro`
- 行业数据点评：优先 `skill-wr-sector-data`

只选择一个主写作 Skill，并把确认后的需求摘要作为后续输入基座。

若用户已经明确指定目标模板 / Skill：

- 优先尊重用户指定
- 本 Skill 仍负责完成需求收敛与摘要确认
- 若信息已足够，则直接轻量交接给指定 Skill
- 不因“已选模板”而跳过需求判断，也不因“走基座”而强行增加追问
- 交接给下游 Skill 时，优先传递结构化 brief，而不是把整段历史原样重复展开

## Token 使用原则

- 本 Skill 应被视为轻量流程编排层，不是第二个完整写作助手
- 信息充分时，使用最短必要输出完成摘要确认或直接交接
- 下游写作 Skill 应消费已确认的 brief，避免重新收集同一批需求
- 除非确有必要，不要把首轮输入、追问文本、摘要文本、大纲说明全文重复拼接给下游

### 5. 按需输出大纲并等待确认

当任务结构较复杂时，在交接给主写作 Skill 前或交接时，先产出可确认的大纲：

- 标题建议
- 一级章节
- 每节核心论点
- 哪些地方需要数据、表格、引用或案例

首轮直接生成的大纲要等待用户确认或修订后，才能进入最终成稿。若大纲来自用户已提交需求表单后的连续流程，可先输出可编辑 `writing_outline`，同时继续按该大纲生成初稿，后续允许用户再改。

当任务属于以下类型时，可跳过大纲，直接进入成稿：

- 通知
- 邀请函
- 简短信函
- 群公告
- 短说明
- 其他篇幅短、结构简单、目标明确的文本

当任务属于以下类型时，优先先出大纲：

- 研究报告
- 政策解读
- 长篇专题稿
- 行业分析
- 领导讲话稿
- 其他结构复杂、篇幅较长、需要先确认组织方式的文本

若运行时支持 `parts[]` 结构化渲染，则输出大纲时必须把**完整大纲正文**包在以下隐藏标记中，方便 Companion 解析为 `writing_outline` 卡片：

```markdown
<!--JLC:WRITING_OUTLINE_START-->
# 标题建议

1. 一级章节
2. 一级章节
3. 一级章节
<!--JLC:WRITING_OUTLINE_END-->
```

要求：

- 标记内只放大纲正文，不要放额外解释
- 标记外可以补一句很短的引导语，例如“下面是建议大纲”
- 若未进入大纲阶段，不要输出该标记

## 需求充分性的判断标准

以下条件满足时，可认为足够进入大纲阶段：

- 用户要写什么：明确
- 写给谁看：明确
- 这篇文稿用来做什么：明确
- 不是完全没有材料支撑，或允许你先按常规结构出框架：明确

篇幅、风格、参考材料如果缺失，可以：

- 使用默认值
- 或在摘要中显式写出假设

但主题、体裁、受众、用途这四项尽量不要缺。

若这四项在首轮输入中已经齐备，即使篇幅、风格、材料尚未完全明确，也应优先：

- 基于已知信息生成摘要
- 对次级缺失项使用默认假设或在摘要中标注待确认
- 不再退回到 `writing_requirements` 表单

### 首轮固定回归样例

以下样例用于约束首轮判断行为。遇到同类输入时，必须按期望阶段推进：

| 编号 | 输入特征 | 期望行为 |
|------|----------|----------|
| F1 | `帮我写一篇面向公司管理层的 2026 年上半年中国成品油市场复盘，要求用于内部经营会，正式分析文风，约 3000 字。` | 信息充分，不输出 `writing_requirements`；直接生成 `writing_requirement_summary` |
| F2 | `帮我写一篇成品油市场分析。` | 信息不足，输出 `writing_requirements`；只追问用途、受众、篇幅或体裁等缺失项 |
| F3 | `把这段研究结论整理成公众号短文：2026 年上半年国内成品油需求温和修复，汽油消费受出行恢复支撑，柴油受基建与物流节奏影响仍有波动，主营炼厂库存压力较去年同期下降。面向产业客户，口吻专业但易读，800 字左右。` | 信息充分且任务简单，不输出 `writing_requirements`；摘要确认后可直接成稿 |

### 连续流固定回归样例

以下样例用于约束“需求卡提交后继续执行”的行为：

| 编号 | 输入与补充 | 期望行为 |
|------|------------|----------|
| T2 | 用户先说 `帮我写一篇成品油市场分析。`，随后补充 `体裁：公众号短文；受众：产业客户；用途：对外传播；篇幅：800 字左右` | 输出 `writing_requirement_summary` 后直接生成短文初稿，并写入 `.md` |
| T3 | 用户先说 `帮我写一份年度行业深度报告。`，随后补充 `体裁：正式研究报告；受众：公司管理层；用途：内部决策；篇幅：6000 字左右` | 输出 `writing_requirement_summary`，再输出带隐藏标记的 `writing_outline`，并写入 `.md` 初稿 |

硬性约束：

- 只要主题、体裁、受众、用途已经能从首轮输入中合理识别，就不要再发需求卡
- 篇幅、风格、参考材料缺失时，优先写入摘要中的默认假设，不作为必然追问理由
- 追问只能补缺失项，不得把已知的主题、体裁、受众、用途改写成问题再问一遍
- 若输出结构化卡片，`part.kind` 必须是 `writing_requirements`；若输出摘要，`part.kind` 必须是 `writing_requirement_summary`

## 问题设计要求

- 只问会显著改变结构、文体或重点的问题
- 问题文案要短，不要长段解释
- 单轮不要混入太多开放题
- 可选择的尽量不要都做成自由文本
- 若用户已上传材料，优先围绕材料用途提问，而不是重新问主题

## 输出要求

- 追问阶段：输出结构化问题或简短编号问题
- 摘要阶段：输出稳定的写作需求摘要；若支持结构化渲染，必须使用 `<!--JLC:WRITING_REQUIREMENT_SUMMARY_START-->` / `<!--JLC:WRITING_REQUIREMENT_SUMMARY_END-->` 包裹完整摘要正文
- 大纲阶段：仅在确有必要时输出可确认的 writing outline；若支持结构化渲染，必须使用 `<!--JLC:WRITING_OUTLINE_START-->` / `<!--JLC:WRITING_OUTLINE_END-->` 包裹大纲正文
- 成稿阶段：交给选中的主写作 Skill 执行，并要求最终产物写入工作区；短文/报告初稿至少写入一个 `.md` 文件
- 文件交付必须真实落盘：只有在实际创建或更新了工作区文件后，才能在最终回复中说“已写入/已完成”并给出路径；如果写入失败，必须明确说明失败原因，禁止声称已生成文件

## 禁止

- 不要在首轮就抛出 7～8 个开放问题
- 不要在首轮信息已经充分时仍输出 `writing_requirements`
- 不要因为用户指定模板 / Skill 就让前端直接绕过基座流程
- 不要因为用户指定模板 / Skill 就机械补发表单
- 不要把基座 Skill 做成与下游 Skill 重复耗 token 的第二套完整写作流程
- 不要在首轮需求未确认时直接生成完整成稿；用户提交需求表单后的连续流程除外
- 不要同时加载多套互相冲突的写作体裁 Skill
- 不要虚构“用户已提供”的材料、篇幅或风格偏好
- 不要虚构已经写入工作区的 `.md` 文件路径
- 不要把追问写成纯闲聊，要让前端可稳定渲染或退化展示
