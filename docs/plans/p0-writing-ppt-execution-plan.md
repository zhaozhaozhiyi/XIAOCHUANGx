## 0. 定位与基线盘点

本计划不是替代 `product-roadmap-v4.md` 中的全局 P0 基线收口。路线图中的 P0 仍指工程基线、构建、桌面壳、对话主链路稳定；本文只处理写作 / PPT 在统一对话壳内的 AI to UI 阶段协议、真实流程回归和交付闭环补强。

### 当前已具备

- 写作 / PPT 已进入统一对话界面，不再走独立表单页
- 写作 / PPT 对话已复用主聊天链路、`parts[]` 与工具执行机制
- 写作已有 `.md` 落盘能力，DOCX 生成到工作区链路已有实现
- PPT 已有 `.html` / `.pptx` 产出能力，PPTX 本地交付物操作链路已有实现
- `skill-writing-base` / `skill-ppt-base` 已作为需求采集基座存在

### 本周真正新增 / 修复

- 收紧首轮追问策略，减少信息充分时误出需求卡
- 补齐 `writing_outline` / `ppt_outline` 的结构化协议与恢复规则
- 打通表单提交后的摘要、大纲 / 页纲、成稿 / 出稿连续阶段
- 为大纲 / 页纲卡增加最小结构化编辑能力
- 把既有文件产出与本地打开 / 定位 / 另存体验纳入真实链路回归，而不是重新设计交付体系

### 不纳入本周硬目标

- 不解决路线图全局 P0 的所有 build / lint / CI 基线问题
- 不全面提升所有 `skill-wr-*` 与 PPT 模板内容质量
- 不做完整富文本编辑器、复杂版本对比、多人协作或云端工作区

## 1. 本周目标

本周目标不是继续扩功能，而是把写作 / PPT 的 AI to UI 主流程从“可演示”收成“可稳定测试、可真实交付回归”。

主闭环：

`首轮输入 → 是否追问 → 需求卡片 → 摘要确认 → 大纲 / 页纲 → 成稿 / 出稿 → 工作区交付`

补充决策：

- 写作不强制每次都经过大纲，大纲是按需阶段
- PPT 默认保留页纲阶段
- 大纲 / 页纲卡不是纯展示卡，P0 支持结构化编辑
- 阶段内容仍由 AI 决定，前端只渲染和提交结构化协议
- 基座与模板之间的交接应优先使用结构化 brief，避免重复传递整段上下文

本周完成后，应达到：

- 写作和 PPT 首轮判断更准确
- 需求卡片不再机械触发
- 卡片提交后能稳定进入摘要、再进入按需大纲 / 页纲
- 大纲 / 页纲卡支持结构化修改，而不是只能靠对话改
- 结果能真实落到工作区并可预览、打开、定位、另存 / 导出副本；已存在能力以回归验证为主
- 验收以真实 Companion / Runtime / `parts[]` 输出为准，不以 mock 为准

## 2. P0.1 首轮追问策略收紧

### 目标

把“只问缺失信息、信息充分时不乱追问”收紧成稳定的 AI 行为要求，而不是转成一套硬编码业务判断。

### 原则

#### 写作模块

AI 在首轮判断为信息已足够进入摘要阶段时，不应再返回 `writing_requirements`：

- 主题
- 体裁
- 受众
- 用途

#### PPT 模块

AI 在首轮判断为信息已足够进入摘要阶段时，不应再返回 `ppt_requirements`：

- 主题
- 场景
- 受众
- 目标

### 执行动作

- 收紧 `skill-writing-base` / `skill-ppt-base` 的首轮追问指令
- 明确“只问缺失信息”与“信息充分时直接进入摘要确认”的行为边界
- 不在前端、Companion、Runtime 中增加业务型首轮裁决代码
- 通过真实链路回归测试验证 AI 是否按预期少问、准问

### 验收标准

- 首轮输入信息完整时，直接进入摘要确认
- 首轮输入信息不足时，出现表单
- 不再出现“明明信息够了还追问”的高频问题
- 问与不问的改善主要来自 Skill / Prompt / AI 行为优化，而不是运行时代码拦截

### 首轮判断测试夹具

首轮判断必须用固定输入回归，避免只凭一次模型输出判断是否通过。

| 编号 | 模块 | 输入样例 | 期望 |
|------|------|----------|------|
| F1 | 写作 | `帮我写一篇面向公司管理层的 2026 年上半年中国成品油市场复盘，要求用于内部经营会，正式分析文风，约 3000 字。` | 不出 `writing_requirements`，直接进入 `writing_requirement_summary` |
| F2 | 写作 | `帮我写一篇成品油市场分析。` | 出 `writing_requirements`，只追问缺失的用途、受众、篇幅或体裁等关键信息 |
| F3 | 写作 | `把这段研究结论整理成公众号短文：2026 年上半年国内成品油需求温和修复，汽油消费受出行恢复支撑，柴油受基建与物流节奏影响仍有波动，主营炼厂库存压力较去年同期下降。面向产业客户，口吻专业但易读，800 字左右。` | 不出 `writing_requirements`，简单任务可摘要后直接成稿 |
| F4 | PPT | `做一份面向客户高层的金联创产品能力介绍 PPT，用于售前沟通，目标是说明数据、研究和交付优势，12 页左右。` | 不出 `ppt_requirements`，直接进入 `ppt_requirement_summary` |
| F5 | PPT | `帮我做个公司介绍 PPT。` | 出 `ppt_requirements`，只追问场景、受众、目标、页数或风格等缺失信息 |
| F6 | PPT | `基于刚才的文稿生成一份 10 页汇报 PPT，面向内部经营会，重点突出市场判断和行动建议。` | 不出 `ppt_requirements`，进入摘要或页纲阶段 |

通过标准：

- 固定样例至少连续 2 轮回归结果一致
- 期望“不出卡”的样例不得返回 `writing_requirements` / `ppt_requirements`
- 期望“出卡”的样例不得追问已在输入中明确给出的字段
- 若模型输出合理但字段名或 `part.kind` 不符合协议，按失败处理

## 3. P0.2 表单后半段主流程打通

### 目标

把表单卡片后的正式主流程打通，而不是停留在“能显示卡片”。
这一阶段的重点不是再加一套写死流程，而是让 AI 输出的阶段协议连续可用、可恢复、可编辑。

### 写作主流程

写作不是单一路径，而是分支型路径：

- 简单 / 短文本任务：`writing_requirements → writing_requirement_summary → 成稿`
- 结构复杂任务：`writing_requirements → writing_requirement_summary → writing_outline → 成稿`
- 首轮信息充分且任务简单：`writing_requirement_summary → 成稿`

结论：

- `writing_outline` 是按需阶段，不是必经阶段
- 前端不能假设每次写作都会出现大纲卡
- 是否进入大纲阶段，由 AI 根据任务类型决定

### PPT 主流程

`ppt_requirements → ppt_requirement_summary → ppt_outline → 交付物`

结论：

- `ppt_outline` 默认保留为主流程阶段
- PPT 前端默认应准备好页纲卡的展示与编辑能力

### 执行动作

- 先扩展 `packages/contracts` 中 `writing_outline` / `ppt_outline` 协议，使其可承载结构化编辑
- 表单提交后，将答案稳定写回会话状态
- Companion 输出中稳定生成 `summary`、可选 `writing_outline`、默认 `ppt_outline` 对应 part
- 前端继续基于 `parts[]` 渲染，不把业务流程写死到页面里
- 刷新页面后，要求能从会话恢复当前阶段
- 阶段协议保持三层分工：
  - `skills/`：定义 AI 在什么条件下输出什么阶段
  - `packages/contracts`：定义阶段 `part.kind` 与结构
  - `companion`：负责解析、持久化、恢复到 `parts[]`

### 协议前置约束

当前仓库中的 `OutlinePart` 仍以 `markdown` 为主，尚不足以稳定支持结构化编辑。

因此，P0.2 必须先完成以下前置动作：

- 为 `writing_outline` / `ppt_outline` 增加结构化数据字段
- 明确写作章节与 PPT 页面在协议中的最小结构
- 明确编辑态、已提交态、最终采用版的存储方式
- 在 Companion 与前端渲染层统一消费同一份结构，而不是各自解析 Markdown

在协议未补齐前，不进入正式“大纲 / 页纲卡结构化编辑”开发。

### 最小结构化协议草案

P0 只要求支持稳定编辑和恢复，不要求覆盖完整富文本表达。

#### `writing_outline`

```ts
type WritingOutlinePart = {
  kind: "writing_outline";
  title: string;
  markdown?: string;
  outline: {
    version: 1;
    source: "ai" | "user";
    committed: boolean;
    sections: Array<{
      id: string;
      title: string;
      bullets: string[];
    }>;
  };
};
```

#### `ppt_outline`

```ts
type PptOutlinePart = {
  kind: "ppt_outline";
  coverTitle: string;
  markdown?: string;
  outline: {
    version: 1;
    source: "ai" | "user";
    committed: boolean;
    slides: Array<{
      id: string;
      title: string;
      bullets: string[];
    }>;
  };
};
```

最小规则：

- `id` 由 Companion 或前端在首次结构化时生成，后续排序、删除、编辑都沿用同一 `id`
- `markdown` 仅作为兼容展示和降级字段，不作为结构化编辑的主数据源
- AI 首次输出保存为 `source: "ai"`；用户提交后保存为 `source: "user"`、`committed: true`
- 恢复会话时优先展示最后一个 `committed: true` 的用户确认版
- 后续继续生成时，Companion 传给 AI 的是最后一次已提交结构，而不是重新解析历史 Markdown

### 大纲 / 页纲卡编辑策略

P0 采用“结构化可编辑”，不做富文本编辑器。

#### 写作大纲卡

若 AI 返回 `writing_outline`，前端支持编辑：

- 标题
- 一级章节顺序
- 每节核心点
- 新增一节
- 删除一节

#### PPT 页纲卡

若 AI 返回 `ppt_outline`，前端支持编辑：

- 封面标题
- 页面顺序
- 每页标题
- 每页 3-5 个要点
- 新增一页
- 删除一页

#### 编辑原则

- 前端只渲染结构和提交结构，不拥有内容判断权
- 编辑后的结构化结果继续作为 AI 下一轮输入
- 不把卡片做成开放富文本编辑器
- 若用户不想在卡片内改，也保留通过对话继续调整的路径

### 编辑回写与恢复规则

为避免前后端各自解释，P0 阶段先固定最小规则：

- AI 首次输出的版本记为“AI 原版”
- 用户在卡片内修改并提交后，生成“用户确认版”
- 会话恢复时，优先展示最后一次已提交的“用户确认版”
- 若用户未提交，仅本地草稿可丢失；已提交版本必须持久化
- 后续继续生成时，AI 应消费“最后一次已提交版本”，而不是回退到最早的 AI 原版
- 是否保留 AI 原版用于对比，可作为 P1 增强，不作为 P0 必需项

### 验收标准

- 表单提交后不丢状态
- 摘要卡片能正常显示
- 写作在“需要大纲”的任务中能出现大纲卡，在“不需要大纲”的任务中能直接进入成稿
- PPT 页纲卡能正常显示
- 大纲 / 页纲卡支持结构化修改并继续推进
- 会话刷新后仍能恢复到正确阶段

## 4. P0.3 工作区交付闭环

### 目标

让写作 / PPT 不只是生成消息，而是真正产生工作区交付物。本周对交付链路以真实回归和缺口修复为主，不重新设计文件体系。

### 执行动作

#### 写作

- 回归真实产出 `.md`
- 回归 DOCX 生成到工作区
- 确认工作区内可预览、可打开、可定位
- 若已有能力失败，按阻塞问题修复

#### PPT

- 回归真实产出 `.html`
- 回归至少 1 次 PPTX 生成或本地打开 / 定位
- 确认工作区内可预览、可打开、可定位
- 交付物卡片与工作区文件树保持一致
- 若 PPTX 无法稳定产出，必须记录明确降级原因和用户可见提示，不得只用 `.html` 静默通过

### 验收标准

- 写作至少成功产出 1 份 `.md`
- 写作至少成功完成 1 次 `.docx` 导出
- PPT 至少成功产出 1 份可预览 `.html`
- PPT 至少成功完成 1 次 `.pptx` 生成或本地打开 / 定位
- 工作区中可见、可打开、可定位

## 7. 建议排期

### 第 1 天

- 完成首轮追问策略收紧
- 明确写作 / PPT 的“信息充分”判断条件，并写入 Skill / PRD 约束

- 先补 outline 结构协议

- 补大纲 / 页纲卡结构化编辑能力
- 优先支持标题、要点、新增、删除、排序；不做富文本和版本对比

- 固化真实测试用例

- 覆盖“写作有大纲 / 无大纲”两条路径
- 若结构化编辑仍不稳定，优先保证协议持久化和恢复，再缩减 UI 交互范围

### 第 5 天

- 集中修复回归问题
- 形成一版可验收版本
- 若 10 条用例无法全部通过，必须标记阻塞用例、降级范围和下一步 owner，不得口头通过

## 8. 开发任务单

以下任务单按五条执行线拆分：`contracts`、`companion`、`web`、`skills`、`验收`。

### 8.1 Contracts

#### C1. 扩展大纲协议

- 为 `writing_outline` / `ppt_outline` 增加结构化字段
- 保留 `markdown` 作为兼容展示字段
- 新增最小结构：
  - 写作：标题、章节数组、每节要点数组
  - PPT：封面标题、页面数组、每页要点数组
- 字段与“最小结构化协议草案”保持一致，至少包含 `version`、`source`、`committed`、稳定 `id`

#### C2. 增加编辑态字段

- 明确大纲 / 页纲卡的 `submitted` / `answer` / `selectedOptions` 之外的编辑字段
- 定义“AI 原版”与“用户确认版”的最小承载方式
- 明确哪些字段用于前端草稿，哪些字段必须持久化
- 用户提交后的结构必须能被 Companion 作为下一轮 AI 输入直接消费

#### C3. 保持兼容

- 不破坏当前只读 `RequirementSummaryCard` / `OutlinePart` 展示链路
- 允许旧的仅 `markdown` 数据继续被渲染
- 仅 `markdown` 旧数据可展示但不可作为结构化编辑的唯一来源；如需编辑，必须先转换为结构化数据

### 8.2 Companion

#### CP1. 需求卡提交后状态持久化

- 写作 / PPT 表单提交后，将结构化答案稳定写回 session
- 刷新页面后能恢复到已提交状态

#### CP2. 摘要与大纲阶段解析

- 继续解析 `writing_requirement_summary` / `ppt_requirement_summary`
- 继续解析 `writing_outline` / `ppt_outline`
- 支持“写作无大纲路径”直接进入成稿

#### CP3. 大纲编辑回写

- 接收前端提交的结构化大纲 / 页纲
- 将“用户确认版”写回消息或会话状态
- 后续继续生成时，优先把最后一次已提交版本提供给 AI
- 明确重复提交的幂等规则，避免同一用户确认版被追加成多个有效版本

#### CP4. 模板交接轻量化

- 指定模板后仍走基座 skill
- 交接给模板时优先传结构化 brief
- 避免把整段历史全文重复拼接给下游
- brief 中必须包含需求摘要、最后一次已提交 outline、目标模板或目标 skill

### 8.3 Web

#### W1. 需求卡稳定提交

- `RequirementsCard` 提交后锁定状态
- 避免重复点击、重复提交
- 刷新页面后正确恢复

#### W2. 摘要卡操作

- 摘要卡支持继续推进
- 摘要卡支持“继续补充 / 修改”入口

#### W3. 大纲 / 页纲卡结构化编辑

- 为写作大纲卡增加结构化编辑 UI
- 为 PPT 页纲卡增加结构化编辑 UI
- 不做富文本编辑器
- 支持新增、删除、调整顺序、编辑标题与要点
- P0 不做 AI 原版 / 用户确认版对比视图；只展示当前有效版本

#### W4. 阶段恢复

- 会话刷新后，仍能恢复 requirements / summary / outline / deliverables 阶段
- 写作无大纲路径也要正确恢复
- 恢复优先级为最后一次已提交用户确认版，其次 AI 原版，其次 markdown 兼容展示

#### W5. 工作区交付体验

- 写作 `.md` 预览与 `.docx` 导出可用
- PPT `.html` / `.pptx` 预览、打开、定位与另存可用
- 交付物卡片与工作区文件树一致
- PPTX 若失败必须暴露用户可见错误或降级提示，不能静默以 HTML 通过

### 8.4 Skills

#### S1. 写作基座 skill

- 收紧首轮追问策略
- 明确写作可跳过大纲
- 明确哪些任务优先直出成稿，哪些任务优先先出大纲
- 指定模板时保持“走基座但轻通过”

#### S2. PPT 基座 skill

- 收紧首轮追问策略
- 保持 PPT 默认页纲阶段
- 指定模板时保持“走基座但轻通过”

#### S3. 模板 / 下游 skill 交接

- 下游写作 / PPT skill 默认消费结构化 brief
- 避免重复需求收集
- 避免与基座 skill 重复消耗 token

### 8.5 验收

#### T1. 首轮判断

- F1 / F3 写作首轮充分，不出卡
- F4 / F6 PPT 首轮充分，不出卡
- F2 / F5 首轮信息不足时，正确出卡
- 固定样例至少连续 2 轮回归结果一致

#### T2. 写作两条主路径

- T2 写作：出卡 → 摘要 → 直接成稿
- T3 写作：出卡 → 摘要 → 大纲 → 成稿

#### T3. PPT 主路径

- T5 PPT：出卡 → 摘要 → 页纲 → 交付物

#### T4. 指定模板路径

- T6 写作指定模板且信息充分：走基座但轻通过
- T7 PPT 指定模板且信息充分：走基座但轻通过
- T8 指定模板但信息不足：先追问，再交接

#### T5. 编辑与恢复

- T9 大纲 / 页纲结构化编辑后可继续推进
- T10 刷新页面后恢复到最后一次已提交版本

#### T6. 工作区交付

- 写作真实产出 `.md`
- 写作成功导出 `.docx`
- PPT 真实产出可预览 `.html`
- PPT 成功完成 1 次 `.pptx` 生成或本地打开 / 定位
- 工作区内可见、可预览、可打开、可定位

## 9. 优先级原则

本周所有开发任务都按下面顺序判断是否值得做：

1. 是否增强了写作 / PPT 主流程稳定性
2. 是否保持 AI 决定阶段内容、前端只承接协议
3. 是否减少了基座与模板之间的重复 token 消耗
4. 是否增强了真实工作区交付能力
5. 是否减少了 mock 依赖、增加了真实验收覆盖
6. 是否不会破坏现有可开发功能

若不满足以上原则，默认延后。

## 10. 完成定义

当以下条件同时满足时，可认为本轮 AI to UI 收口完成：

- 写作 / PPT 首轮追问行为符合预期
- AI to UI 卡片只在真正缺信息时触发
- 卡片后的摘要与按需大纲 / 页纲流程稳定
- 写作“有大纲 / 无大纲”两条路径都可用
- 指定模板时仍走基座，但能轻通过
- 大纲 / 页纲卡支持结构化编辑
- 交付物真实落到工作区
- 至少 10 条真实验收用例全部通过
- 若存在未通过用例，必须记录阻塞原因、降级范围、下一步 owner，不能视为完成
