## 1. 路线总览

> **当前明确决策：** 小窗当前导航六模块全部可见：对话 / 写作 / PPT / 3D绘图 / 视频 / 推演。翻译、会议纪要放到后续版本候选；知识库明确不需要，不再推进。  
> **平台边界：** 当前 `0.1.0-alpha` 以 Desktop + 本地 Companion + 本地文件夹工作区为准；Web 在线沙箱化放到下一大版本。  
> 平台下一阶段围绕一个核心闭环展开：  
> **对话研究 → 形成文稿 → 生成 PPT → 文件落工作区**

| 阶段 | 目标 | 核心结果 |
|------|------|----------|
| **P0 基线收口** | 让平台从“可演示”变成“可持续开发” | 构建稳定、桌面壳稳定、对话主链路稳定 |
| **P1 对话成型** | 把对话做成真正可交付的研究入口 | 研究流程、工作区产出、会话体验、可视化增强 |
| **P2 写作收口** | 形成第一个正式内容生产模块 | 大纲确认、成稿、导出、历史复用 |
| **P3 PPT 收口** | 让研究成果可直接转成汇报材料 | 大纲生成、交付物预览、PPTX 导出 |
| **P4 子线模块收口** | 3D / 视频 / 推演按各自成熟度推进 | 3D M1 发布闸门、视频 Remotion 闭环、推演画布与快照 |
| **P5 Web 在线沙箱与多人协作** | 进入 Web Sandbox 阶段 | Web 在线工作区、云端运行时、多用户后台、跨设备、协作 |

## 2. 当前判断

### 2.1 已经具备的基础

- 产品主线已经成立：对话、写作、PPT 围绕统一对话主壳组织；3D、视频、推演也已作为同级入口接入共享会话壳
- 平台底座已经存在：`web + companion + desktop + runtime-core`
- 工作区模型已经成立：所有内容都围绕 `projectId`
- 模块策略是对的：写作 / PPT 不是独立系统，而是共享执行主干

### 2.2 当前最大问题

- 基线构建已经可过，但仍需要把 `web build` 的 Turbopack NFT tracing warning、lint 基线与最小 CI 固化为常态闸门
- 对话已经完成 Alpha 有条件收口，但“研究结果如何稳定沉淀成文件并复用到写作 / PPT”的链路仍需继续打磨
- 写作 / PPT 主链路与 T9/T10 已通过真实 smoke / E2E，但本地工作区交付物体验（生成、打开、定位、另存 / 导出副本）、历史侧栏与更多真实多轮场景仍需收紧

### 2.3 这意味着什么

下一步不应该再新增第七个业务入口，而应该纵向打穿现有六模块中已承诺的闭环与发布闸门。

## 3. P0 基线收口

### 3.1 目标

先把平台做成“能稳定迭代的产品工程”，而不是“功能越来越多但越来越脆的原型工程”。

### 3.2 核心任务

- 固化 `web build`、TypeScript、Companion / Desktop typecheck 为最小 CI 闸门
- 收敛 `lint` 规则问题，至少先清掉核心页面、运行时和桌面壳相关错误
- 清理无效 mock、历史生成物、演示文件对工程检查的干扰
- 建立最小 CI 基线：构建、关键 lint、关键 smoke

### 3.3 验收标准

- `web build` 可通过，且已知 warning 有 owner 与处理计划
- 对话主页面和桌面壳主路径无明显回归
- 关键链路可以通过 smoke 验证

---

## 4. P1 对话成型

### 4.1 目标

把对话从“聊天入口”做成“研究生产入口”。

### 4.2 核心任务

- 稳定 `parts[]` 展示节奏
- 强化深度模式的研究流程表达
- 提升工作区联动：文件产出、深链打开、产出可视
- 强化会话管理：历史、状态、继续对话、停止生成
- 补齐数据源 / 图表 / 多信源 / 溯源能力

### 4.3 验收标准

- 用户可以只靠对话完成一次完整研究任务
- 研究结果可以稳定落到工作区
- 研究结果可以继续被写作 / PPT 复用

---

## 5. P2 写作收口

### 5.1 目标

写作成为第一个“正式可交付”的业务模块。

### 5.2 核心任务

- 明确写作模板入口与默认策略
- 强化大纲确认与回改流程
- 完整打通 Markdown → DOCX 本地生成与工作区打开 / 定位 / 另存
- 提升 `skill-wr-*` 内容质量
- 完善历史复用：从对话研究结果进入写作

### 5.3 验收标准

- 用户可以从空白需求或已有研究结论生成完整文稿
- 可导出可交付的 Word 文档
- 文稿在工作区中可管理、可复用

---

## 6. P3 PPT 收口

### 6.1 目标

让“小窗”从研究与写作工具，进化为“研究到汇报”的工作台。

### 6.2 核心任务

- 明确主题生成与从文稿生成两条路径
- 提升大纲生成质量
- 强化 HTML / PPTX 交付物预览
- 补齐本地打开、定位、另存 / 导出副本体验
- 明确图表进入幻灯片的规则

### 6.3 验收标准

- 用户可以从主题或文稿得到可用的 PPT 初稿
- PPTX 导出稳定可用
- 交付物在工作区中可浏览、可复用

---

## 7. P4 子线模块收口

### 7.1 目标

六模块保持可见，但按成熟度分层承诺：3D绘图优先完成 M1 发布闸门，视频与推演先补齐真实闭环的协议、工具链与 smoke。

### 7.2 核心任务

- 3D：OpenSCAD Runtime 制品库、许可证材料、WASM 预览开关、导出失败重试。
- 视频：`vp_*` parts、Remotion 项目生成、渲染 BFF / Companion 工具、MP4 预览下载。
- 推演：`simulation_*` parts、React Flow 画布、Round 快照、报告导出。

### 7.3 验收标准

- 3D M1 能从需求生成可编辑 SCAD，并完成预览与导出硬闸门。
- 视频至少完成从 brief 到 Remotion 项目 / 可预览产物的真实 smoke。
- 推演至少完成需求卡、画布数据、Round 快照与报告文件的真实 smoke。

---

## 8. P5 企业化阶段

### 8.1 目标

在六模块当前承诺范围稳定后，再进入多人协作与企业部署阶段。

### 8.2 核心任务

- 启用 Nest 多用户后台
- 建设 Web 在线沙箱工作区与云端运行时
- 补齐跨设备会话与权限能力
- 建立企业分发、审计、配置能力

### 8.3 不建议提前做的事

- 在 P0-P4 阶段过早投入多人协作
- 过早启动会议纪要模块，或重启知识库模块
- 在底座未稳时继续扩横向能力

## 9. 优先级原则

后续所有需求都建议按下面顺序决策：

1. 是否增强了 **对话研究 → 文件产出** 的主链路
2. 是否增强了 **写作 / PPT / 3D / 视频 / 推演** 中已经可见模块的闭环
3. 是否会破坏当前桌面优先、本地优先的产品形态
4. 是否会把团队重新拉回“重后台、大而全”的路线

如果答案偏离这四条，建议默认不做。
