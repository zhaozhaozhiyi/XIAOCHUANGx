## 2. 核心产品判断

### 2.1 当前主线

平台的核心闭环不是继续横向加模块，而是纵向打通：

**对话研究 -> 形成文稿 -> 生成 PPT -> 文件落工作区**

### 2.2 实现原则

后续实现必须遵守以下原则：

1. 不影响现有开发中的功能，不删减当前可见能力。
2. 不重写现有主架构，基于现有 `web + companion + runtime-core + desktop` 继续增强。
3. 统一使用“会话壳 + 模块参数化”的方式，而不是为每个模块各造一套系统。
4. 所有正式交付物最终都要回到工作区管理。
5. 业务页负责“产出提示与快捷入口”，Desktop 工作区负责“正式预览、打开、定位、导出 / 另存副本”；Web Sandbox 可补充下载。
6. 涉及数据流、接口流、SSE 事件流、文件落盘流的验收与回归，必须走真实链路测试，不能只用 mock 结果判定完成。

### 2.3 数据流测试规范

后续所有模块实现统一遵守以下测试规则：

1. `mock` 仅用于前端占位、交互联调、协议演示与开发期快速验证。
2. 任何“已完成”“可验收”“可上线”的判断，必须补充真实数据流测试，不得只依据 mock 截图或假数据演示。
3. 真实测试至少覆盖：
   - 真实请求进入 `web` / BFF
   - 真实 Companion / Runtime 返回
   - 真实 SSE 事件到前端渲染
   - 真实工作区文件写入、预览、打开、定位或导出 / 另存副本
4. 若当前阶段只能先做 mock，文档和汇报中必须明确标注“仅完成 mock 联调，未完成真实链路验收”。
5. 后续新增 `parts[]`、导出、预览、工作区文件能力时，默认要求同时提供真实链路测试记录；下载链路仅在 Web Sandbox 或兼容文件流场景单独验收。

## 4. 总体实现策略

### 4.1 产品结构

采用“一个会话主壳，六个业务表面”的结构：

- `chat`
- `writing`
- `ppt`
- `3d`
- `video`
- `simulation`

六个模块继续共用统一会话组件和执行主链路，不拆成六套独立页面框架。本文后续章节仍以对话 / 写作 / PPT 主线为重点，3D / 视频 / 推演以子线文档作为验收权威。

### 4.2 模块职责

| 模块 | 主要职责 | 输出物 |
|------|----------|--------|
| 对话 | 研究、问答、资料整理、任务分发 | 结论、引用、文件草稿、交付物卡片 |
| 写作 | 基于需求或研究结果产出文稿 | Markdown、DOCX |
| PPT | 基于主题或文稿产出演示文稿 | HTML、PPTX |

### 4.3 平台能力边界

平台负责：

- 会话编排
- Agent 运行
- SSE 流式过程
- 工作区文件落盘
- 文件预览、打开、定位、导出 / 另存副本入口

模块负责：

- 各自的模板 / Skill / 输出格式
- 各自的结果卡片和操作按钮
- 各自的导出规则

## 5. 前端设计方案

## 5.1 页面总结构

四个模块继续复用统一页面骨架：

1. 顶部轻导航
2. 中间会话时间线
3. 底部输入区
4. 右侧工作区

### 5.2 顶部区域

顶部展示：

- 当前模块标题
- 当前会话状态
- 当前工作区名称
- Agent / 模式信息

不增加复杂二级导航，不改变当前整体信息架构。

### 5.3 中间会话时间线

会话区继续按 `parts[]` 交错展示，不将“结果”和“过程”拆为两套视图。

主要分块包括：

- `summary`
- `tool_batch`
- `file_read`
- `file_edit`
- `reasoning`
- `artifact`
- `deliverables`
- `clarification`
- `error`

呈现原则：

- 结果优先可读
- 过程默认可折叠
- 当前轮次保持流式可见
- 交付物以卡片形式突出

对于需要先补充结构化信息的模块，时间线中应支持 **AI to UI** 类型的表单卡与摘要卡，而不是只支持纯文本追问。

### 5.4 底部输入区

统一使用 Composer，但按模块显示不同控件：

| 模块 | Composer 差异项 |
|------|-----------------|
| 对话 | `fast / deep` 模式、附件、Agent 选择 |
| 写作 | 模板选择、需求采集入口、附件、Agent 选择 |
| PPT | deck 模板选择、需求采集入口、附件、Agent 选择 |

### 5.5 结果卡片设计

业务页中的结果卡片仅承担三个职责：

- 告诉用户生成了什么
- 提供快捷跳转
- 提供继续处理入口

不在业务页中承载完整的文件管理能力。

交付物卡片建议保留或新增以下动作：

- `在工作区预览`
- `打开`
- `在文件夹中显示`
- `继续生成文稿`
- `继续生成 PPT`

### 5.6 AI to UI 表单协议

对于写作、PPT 这类高交付密度模块，不建议完全依赖自由文本追问，而应支持 **AI to UI** 协议：

1. 用户先输入一句需求
2. 模块默认基座 Skill 判断信息是否充分
3. 若信息不足，Skill 返回结构化问题
4. 前端按协议渲染为问题卡
5. 用户提交回答
6. Skill 输出需求摘要并请求确认
7. 确认后进入大纲 / 页纲 / 生成

建议前端统一支持以下问题类型：

- `text`
- `textarea`
- `single_select`
- `multi_select`
- `file_pick`
- `file_upload`

建议统一支持以下 `part.kind`：

- `writing_requirements`
- `writing_requirement_summary`
- `writing_outline`
- `ppt_requirements`
- `ppt_requirement_summary`
- `ppt_outline`

## 6. 工作区设计方案

### 6.1 工作区定位

工作区是所有正式交付物的统一管理中心，负责：

- 文件树浏览
- 文件正式预览
- 文件打开
- 文件导出 / 另存副本
- 文件定位
- 文件复用

业务页中的卡片不再承担完整预览 / 文件管理职责，而是把用户引导回工作区。

### 6.2 工作区预览原则

工作区预览区继续复用现有 [FileViewer.tsx](../../web/src/components/workspace/FileViewer.tsx)，并在其上增强统一操作栏。

当前已具备预览基础：

- Markdown 预览 / 源码
- HTML 渲染 / 源码
- PPTX 预览
- PDF 预览
- DOCX 预览
- 图片预览

### 6.3 工作区工具栏

建议将正式文件操作统一放入工作区文件预览区顶部工具栏：

- 打开原文件
- 导出 / 另存副本
- 刷新预览
- 在浏览器中打开
- 在文件夹中显示

### 6.4 不同文件类型的工作区操作

| 文件类型 | 预览方式 | 操作 |
|----------|----------|------|
| `.md` | Preview / Source | 生成 DOCX、复制路径、在文件夹中显示 |
| `.html` | Render / Code | 浏览器打开、复制路径、在文件夹中显示 |
| `.pptx` | 内置预览 | 系统打开、复制路径、在文件夹中显示、另存副本 |
| `.docx` | 内置预览 | 系统打开、复制路径、在文件夹中显示、另存副本 |
| `.pdf` | 内置预览 | 系统打开、复制路径、在文件夹中显示、另存副本 |
| 图片 | 图片预览 | 系统打开、复制路径、在文件夹中显示、另存副本 |

### 6.5 工作区与交付物卡片协作

标准交互路径如下：

1. 用户在对话或模块页生成结果
2. 页面中出现交付物卡片
3. 用户点击 `在工作区预览`
4. 工作区自动展开并定位到对应文件
5. 右侧预览区展示内容
6. 用户在工作区执行打开、定位、导出 / 另存副本或继续处理

## 7. 对话模块实现方案

### 7.1 模块目标

对话模块从“聊天入口”升级为“研究生产入口”。

### 7.2 前端能力

对话模块前端重点建设：

- 稳定 `parts[]` 时间线展示
- 优化深度模式研究节奏
- 强化交付物卡片
- 强化会话状态展示
- 打通与工作区的文件联动

### 7.3 结果展示

对话模块的结果建议统一为：

- 结论摘要
- 引用 / 来源
- 过程折叠块
- 交付物卡片
- 继续处理入口

### 7.4 继续处理入口

在对话结果中直接提供：

- 基于本结果生成写作文稿
- 基于本结果生成 PPT

这类动作本质是把已有产物或结论传给下一个模块，而不是新建另一套页面逻辑。

## 8. 写作模块实现方案

### 8.1 模块目标

写作模块成为第一个正式可交付内容模块。

### 8.2 页面与交互

写作页继续复用会话壳，但建议与 PPT 一样，采用“默认基座 Skill 先收敛需求，再进入成稿”的方式。

标准流程：

1. 输入主题或选择已有研究结果
2. 默认基座 Skill 判断信息是否充分
3. 信息不足时返回结构化问题
4. 用户提交回答
5. 生成需求摘要并确认
6. 输出大纲
7. 用户确认或补充要求
8. 生成正文
9. 落 Markdown
10. 在工作区预览与导出 DOCX

### 8.3 默认基座 Skill 职责

写作模块也建议拆成两层：

- 默认基座 Skill：负责需求采集、AI to UI 追问、需求摘要确认
- 具体写作 Skill：负责文体、结构、输出规范

默认基座 Skill 应负责：

- 判断写作任务信息是否充分
- 信息不足时输出结构化问题
- 汇总用户回答为写作需求摘要
- 在摘要确认后进入大纲阶段
- 将确认后的 brief 交给具体写作 Skill

### 8.4 写作 AI to UI 协议

写作模块的 AI to UI 逻辑应与 PPT 保持一致，只是问题内容不同。

典型问题维度包括：

- 文稿主题
- 文稿类型
- 目标受众
- 目标用途
- 篇幅要求
- 风格要求
- 是否有参考材料
- 必须保留的观点、数据、口径

建议新增的 `part.kind`：

- `writing_requirements`
- `writing_requirement_summary`
- `writing_outline`

### 8.5 输出物

写作模块统一输出：

- Markdown 正文
- DOCX 导出文件

### 8.6 页面卡片

写作页结果卡片展示：

- 当前文稿名称
- 文稿状态
- 在工作区预览
- 导出 Word

## 9. PPT 模块实现方案

### 9.1 模块目标

PPT 模块承接研究与文稿，形成可汇报的演示交付物。

### 9.2 页面与交互

PPT 页建议支持两种进入方式：

- 从主题直接生成
- 从已有文稿继续生成

但与普通对话不同，PPT 主流程不建议让用户直接一句话进入生成，而是参考 Open Design 的工作方式，先进行一轮“需求采集”，再进入大纲与生成阶段。

这套逻辑不建议写死在前端页面中，而应写进 **PPT 模块默认基座 skill**。也就是说，PPT 默认基座 skill 负责判断信息是否充分、何时发起追问、何时生成需求摘要、何时允许进入页纲阶段。

标准流程：

1. 输入主题或选择文稿
2. 先进入需求采集阶段
3. 用户补充必要信息
4. 输出页纲
5. 用户确认页数 / 风格 / 结构
6. 生成 HTML / PPTX
7. 在工作区预览、打开、定位或另存

### 9.3 默认基座 skill 职责

PPT 模块建议引入一个默认基座 skill，负责通用流程，而不是把“先追问再生成”的逻辑分散写在每个风格 skill 中。

建议职责如下：

- 接收用户初始需求
- 判断当前信息是否足够开始做 PPT
- 信息不足时返回结构化问题
- 接收用户回答并汇总为需求摘要
- 驱动用户确认需求摘要
- 确认后再进入页纲生成
- 将需求摘要传给具体 PPT 风格 skill

建议定位：

- 默认基座 skill 负责“流程”
- 具体 PPT 模版 skill 负责“风格、结构、内容偏好、输出规范”

这样可以保证不同 PPT 模版都遵守同样的要求收集逻辑，同时保留不同模版的内容差异。

### 9.4 需求采集阶段

PPT 是高交付密度任务，前置要求信息如果不收够，后续返工会很多。因此建议在 PPT 模块中加入一个轻量“要求收集卡”，但仍放在现有会话流中，不做独立复杂表单页。

这里不是前端预先写死一整张表单，而是采用 **AI to UI** 模式：

1. 用户先输入一句需求
2. 默认基座 skill 判断缺少哪些关键信息
3. skill 返回结构化问题给前端
4. 前端根据 schema 渲染问题卡片
5. 用户填写并提交
6. skill 生成需求摘要并请求确认

默认基座 skill 通常需要补齐的信息包括：

- 演示主题
- 使用场景
- 目标受众
- 汇报目标
- 页数范围
- 风格偏好
- 是否有参考材料
- 是否有必须保留的结论 / 图表 / 数据

其中前四项建议作为必填信息，其他项可选。

### 9.5 要求收集卡设计

要求收集卡不是静态页面表单，而是前端对 AI 返回问题的结构化渲染结果。

前端职责：

- 渲染 AI 返回的问题
- 支持单选、多选、文本、文件引用等问题类型
- 展示问题说明、默认值、校验状态
- 提交回答并继续当前 run

问题不是完全写死，但建议支持一组稳定的问题控件类型：

| 问题类型 | 说明 |
|----------|------|
| `text` | 单行文本 |
| `textarea` | 多行描述 |
| `single_select` | 单选 |
| `multi_select` | 多选 |
| `file_pick` | 从工作区选择文件 |
| `file_upload` | 上传附件 |

AI 返回的典型问题，可能会围绕以下信息生成：

| 信息维度 | 说明 |
|----------|------|
| 演示主题 | 这次 PPT 的主题是什么 |
| 使用场景 | 路演 / 内部汇报 / 客户提案 / 周报 / 培训等 |
| 目标受众 | 领导、客户、投资人、内部团队等 |
| 汇报目标 | 想说服对方什么、传达什么 |
| 页数范围 | 5 页内 / 5-10 页 / 10-20 页 / 20 页以上 |
| 风格偏好 | 专业、简洁、数据型、路演型、品牌型 |
| 参考材料 | 文稿、报告、表格、历史 PPT |
| 必须保留内容 | 关键数据、图表、结论、口径 |

交互原则：

- 默认先由 AI 判断是否需要追问，不直接生成
- 若用户一句话信息很完整，可少问或不问
- 若信息不足，优先返回结构化问题，而不是反复自然语言追问
- 问题卡提交后生成一份“需求摘要”，供用户确认

### 9.6 AI to UI 协议建议

默认基座 skill 返回给前端的问题建议遵循结构化协议。

建议问题 schema 至少包含：

- `id`
- `label`
- `type`
- `required`
- `placeholder`
- `options`
- `description`

建议新增的 `part.kind` 已在“前端 AI to UI 表单协议”中统一定义。

其中：

- `ppt_requirements`：AI 返回的问题卡
- `ppt_requirement_summary`：AI 汇总后的需求摘要卡
- `ppt_outline`：页纲卡

### 9.7 需求确认与状态设计

参考 Open Design 的状态覆盖原则，PPT 要求收集卡必须覆盖完整状态，而不能只画正常填写态。

必须具备的状态：

- 未填写
- 填写中
- 校验错误
- 提交中
- 已生成需求摘要
- 已确认，可进入页纲生成

错误与校验原则：

- 必填项建议在 `blur` 后校验，不要首字输入即报错
- 错误提示要明确告诉用户缺了什么，而不是只显示“请完善信息”
- 提交失败后保留已填内容
- 若 15 秒以上无响应，给出“生成需求摘要耗时较长”的状态提示

### 9.8 页纲生成阶段

需求确认后，才进入页纲生成阶段。

页纲阶段建议展示：

- 推荐封面标题
- 建议页数
- 每页标题
- 每页核心信息
- 哪些页需要图表 / 数据 / 案例

用户在这一步可以：

- 调整页数
- 调整结构
- 增删页面
- 指定某页重点展开

### 9.9 交付物生成阶段

在页纲确认后，Agent 再正式生成交付物：

- HTML 预览文件
- PPTX 文件

页面中仍通过交付物卡片提示结果，但正式预览、打开、定位、导出 / 另存副本统一回到工作区。

### 9.10 输出物

PPT 模块统一输出：

- HTML 预览文件
- PPTX 文件

### 9.11 页面卡片

PPT 页结果卡片展示：

- 演示名称
- 交付类型
- 在工作区预览
- 打开 PPTX / HTML
- 在文件夹中显示

### 9.12 前端实现建议

在不改变现有会话主壳的前提下，PPT 模块建议新增以下前端能力：

- `PptRequirementsCard`
- `PptRequirementSummaryCard`
- `PptOutlineCard`
- `PptGenerateActionBar`

这些组件都作为 `parts[]` 中的新型结果块进入时间线，而不是另做独立全屏流程页。

### 9.13 后端与编排建议

PPT 模块运行时建议分为三个逻辑阶段：

1. 要求收集
2. 页纲生成
3. 交付物生成

Companion / Runtime 不需要拆三套接口，仍可复用统一 Run，只是在 `binding` 或 `part.append` 事件中表达当前阶段。

建议新增的阶段状态：

- `requirements_collecting`
- `requirements_confirmed`
- `outline_generating`
- `outline_confirmed`
- `ppt_generating`

后端能力上优先补充：

- 保存 PPT 需求摘要
- 保存页纲结构
- 将需求摘要和页纲作为生成 PPT 的显式输入

这样后续多轮修改时，Agent 不需要每轮重新猜用户真实意图。

## 11. 后端与接口方案

### 11.1 总体原则

后端继续保持“统一 Run 编排 + 模块参数化”。

不建议为每个可见模块拆一套独立执行服务。

### 11.2 当前主链路

当前主链路继续保持：

1. Web `POST /api/chat`
2. 转发 Companion `POST /v1/runs`
3. Companion 调用 `runtime-core`
4. Agent 产出 SSE 事件与工作区文件
5. Web 用 `parts[]` 渲染过程与结果

### 11.3 现有关键接口

当前继续复用的主接口：

- `POST /api/chat`
- `GET /api/runs/:runId`
- `GET /api/runs/:runId/events`
- `POST /v1/runs/:runId/clarification`
- `POST /v1/runs/:runId/cancel`
- `POST /api/writing/export-docx`
- `POST /api/ppt/download`
- `GET /api/workspace/file`

### 11.4 建议新增接口

建议新增或补齐以下轻量接口：

| 接口 | 作用 |
|------|------|
| `GET /api/sessions/:id/artifacts` | 返回当前会话产物列表 |
| `POST /api/chat/continue-with-artifact` | 将已有文件作为下一模块输入 |
| `GET /api/workspace/tree` | 工作区树与预览侧协同 |

### 11.5 接口职责划分

本地文件操作：

- Desktop 主路径统一由工作区发起
- 适用于 `pdf / docx / pptx / html / md / png` 等原文件的系统打开、文件夹定位、复制路径、另存 / 导出副本

导出类接口：

- 仍归模块接口负责
- 但入口统一在工作区工具栏中

示例：

- Markdown 生成 DOCX -> `/api/writing/export-docx`
- HTML / PPTX 打开 / 定位 -> 工作区文件操作；`/api/ppt/download` 仅作为 Web Sandbox 或兼容文件流入口

## 13. 页面交互设计

### 13.1 统一交互原则

用户始终在一个会话流里工作，不需要重新学习多套页面。

### 13.2 标准交互闭环

统一闭环如下：

1. 用户提出任务
2. Agent 流式执行并输出结果
3. 页面用卡片提示“已生成什么”

- 重点展示研究结论、来源、交付物、继续处理

写作页：

- 重点展示大纲、正文、导出状态、工作区预览入口

PPT 页：

- 重点展示页纲、交付物、工作区预览入口、本地打开 / 定位动作

## 15. 本阶段明确结论

1. 当前六模块导航全部可见：对话、写作、PPT、3D绘图、视频、推演。

4. 不改现有主架构，只做增量增强。
5. 不删现有功能，不影响当前开发中的可用能力。
6. 业务页负责产出提示与快捷入口。
7. Desktop 工作区负责正式预览、打开、定位、导出 / 另存副本；下载属于 Web Sandbox 或兼容文件流。
8. 下一阶段优先打通“对话研究 -> 文件产出 -> 模块流转”的主链路，并为 3D / 视频 / 推演分别补齐真实 smoke。
