## 1. 版本定位

`0.1.1-alpha` 是 `0.1.0-alpha` 之后、`0.2.0-beta` 之前的小优化版本。

本版本不新增大的业务模块，不改变当前六模块导航，也不提前进入 Web Sandbox / 多用户后台。重点用于收纳 Desktop Alpha 阶段暴露出的轻量体验修复、左侧栏工作流优化、工作区项目管理小能力和可快速验证的交互补强。

---

## 2. 范围原则

### 2.1 纳入范围

- 不改变主架构的小交互优化。
- 不影响发布节奏的局部功能补齐。
- Desktop 本地工作区、左侧栏、项目列表、会话入口等高频路径体验优化。
- 能通过手工验收或轻量 E2E / smoke 验证的需求。

### 2.2 不纳入范围

- 写作 / PPT 的完整 Beta 收口能力，继续归入 `0.2.0-beta`。
- Web Sandbox、多用户后台、云端 Runtime、多人协作。
- 新增第七个一级业务模块。
- 大范围信息架构重构。

## 4. 需求详情

### A011-PROJ-001 左侧项目右键菜单

#### 背景

当前左侧栏展示项目后，用户需要更快地对某个项目执行本地文件夹定位或从列表中移除。参考现有截图中的项目菜单形态，项目行应支持右键呼出上下文菜单。

#### 触发位置

- 左侧栏项目列表中的单个项目行。
- 用户在项目行上点击右键 / 触控板辅助点击时打开菜单。

#### 菜单项

| 菜单项 | 行为 | 说明 |
|--------|------|------|
| 打开文件夹 | 在系统文件管理器中打开该项目绑定的本地文件夹 | macOS 文案可显示为“在 Finder 中显示”；Windows 可显示为“在资源管理器中显示”；产品文档统一称“打开文件夹” |
| 移除 | 将该项目从左侧项目列表中移除 | 仅移除项目绑定 / 列表记录，不删除本地文件夹和其中的文件 |

#### 交互规则

- 右键菜单只作用于当前被点击的项目。
- 菜单打开时，应高亮或标记当前项目行，避免用户误判操作对象。
- 点击菜单外部、按 `Esc`、切换路由或窗口失焦时关闭菜单。
- “打开文件夹”执行后菜单关闭。
- “移除”应有确认机制，避免误移除。
- 若移除的是当前正在使用的项目，应切换到安全的默认状态：
  - 若仍有其他项目，切换到最近使用的可用项目或平台默认项目。
  - 若没有其他项目，进入默认本地工作区或引导用户选择文件夹。

#### 数据与安全边界

- “移除”不得删除用户本地磁盘文件。
- “移除”不得清空历史会话数据，除非后续另有明确需求。
- 项目移除后，该项目不再出现在左侧项目列表和项目选择器中。
- 若用户后续重新导入同一文件夹，应按现有导入逻辑恢复为一个可用项目。

#### 异常状态

| 场景 | 期望表现 |
|------|----------|
| 项目本地路径不存在 | “打开文件夹”提示路径不可用，并保留项目记录 |
| 系统打开文件夹失败 | 显示轻量错误提示，菜单关闭 |
| 移除请求失败 | 保留项目，显示轻量错误提示 |
| 项目正在执行任务 | 默认允许移除列表记录，但不得中断已有 Agent Run；如实现存在风险，可先禁用“移除”并提示任务进行中 |

#### 验收标准

- 在左侧项目行右键后，可以看到“打开文件夹”和“移除”两个操作。
- 点击“打开文件夹”后，系统文件管理器能定位到该项目本地文件夹。
- 点击“移除”并确认后，该项目从左侧项目列表消失。
- 移除项目不会删除本地文件夹和文件。
- 刷新或重启应用后，被移除项目仍不出现在项目列表中。
- 当前项目被移除后，应用不会停留在无效项目状态。

#### 实现备注

`0.1.1-alpha` 已接入项目行右键菜单与悬浮更多按钮：真实用户项目支持“打开文件夹”和“移除”。“打开文件夹”通过 Desktop IPC 按项目 ID 解析并打开项目根目录；“移除”写入本地隐藏项目列表，仅从侧栏和项目选择器移除项目绑定，不删除磁盘文件，也不清空会话数据。默认工作文件夹分组不提供移除，并继续固定在侧栏最后。

### A011-PROJ-002 统一默认工作文件夹与默认创建项目

#### 背景

产品语义上，小窗里的正式任务都应该属于某个文件夹项目。用户没有手动选择已有项目，并不代表任务没有项目；系统应在默认工作文件夹下自动创建任务文件夹，并把该任务文件夹作为项目使用。

当前左侧栏同时出现“默认工作区（XIAOCHUANG）”与“默认工作文件夹（XIAOCHUANG）”两种默认概念，容易让用户误以为存在两个默认文件夹。需要统一为一个清晰口径。

#### 产品口径

- “没有选择项目”只允许作为首条消息发送前的草稿态。
- 一旦用户发起正式任务，系统必须自动创建默认文件夹项目，并分配 `projectId`。
- 默认创建项目与用户手动选择项目都属于“项目”，只是来源不同：
  - 用户手动选择项目：`bindingSource = user_picked`
  - 系统默认创建项目：`bindingSource = platform_default`
- 侧栏不应长期展示“无项目”任务分组。
- 用户视角只保留一个默认根概念：默认工作文件夹（XIAOCHUANG）。

#### 侧栏展示规则

| 场景 | 展示规则 |
|------|----------|
| 用户手动选择项目 | 按项目名称展示，如“公众号”“同步空间” |
| 系统默认创建项目 | 展示在默认工作文件夹语义下，不再额外制造“默认工作区”概念 |
| 首条消息前草稿态 | 可以临时显示“将使用默认工作文件夹” |
| 历史兼容的 `projectId = none` 会话 | 不应作为一个与默认项目并列的长期主分组；应迁移、合并或弱化为兼容态 |

默认工作文件夹分组必须永远排在侧栏项目列表最后，并通过分隔线与上方用户手动选择 / 导入的项目区分。

#### 数据规则

- 正式会话记录应尽快写入真实 `projectId`。
- `projectId = none` 不应成为任务完成后的稳定状态。
- 默认创建的项目必须可用于文件树、`@` 文件提及、交付物打开、继续对话和历史归档。
- 默认创建的项目不得与用户手动选择的项目混淆；两者可在数据层保留 `bindingSource` 区分。

#### 验收标准

- 用户不选择项目直接发起任务后，该任务会拥有真实 `projectId`。
- 左侧栏不再同时出现“默认工作区（XIAOCHUANG）”和“默认工作文件夹（XIAOCHUANG）”两个并列默认分组。
- 默认创建的任务项目可以继续打开、查看文件树、继续对话。
- 历史中的无项目会话不会干扰当前默认工作文件夹语义。
- 文案统一使用“默认工作文件夹（XIAOCHUANG）”作为用户可见默认根概念。
- 默认工作文件夹分组固定在项目列表最后；当上方存在其他项目时，默认工作文件夹上方显示分隔线。

#### 实现备注

`0.1.1-alpha` 已先完成侧栏语义收敛：`platform_default` 默认创建项目与历史兼容的 `projectId = none` 会话统一归入“默认工作文件夹（XIAOCHUANG）”分组，不再并列展示两个默认概念。新任务仍由 `ensure-default-task-project` 物化为真实 `projectId`；历史 `none` 会话仅作为兼容态展示。

### A011-CHAT-001 收敛执行结束图标密度

#### 背景

当前对话执行时间线中，工具运行、文件写入、思考过程和正文输出之间会出现多个连续的“结束 / 完成”状态图标。截图中多个绿色勾选图标沿时间线连续出现，但它们表达的信息相近，容易形成视觉噪声，让用户误以为每个空白节点都是一个重要步骤。

#### 问题描述

- 同一轮 Agent 回复内，多个子事件完成后都单独显示结束图标。
- 文件产物项前后也可能出现额外完成图标，导致“文件图标 + 完成勾 + 文件图标 + 完成勾”重复感明显。
- “思考过程”前后的完成图标与正文开始前的完成图标相邻，信息价值较低。
- 用户真正需要关注的是当前是否仍在运行、是否失败、关键工具 / 文件产物是否完成，而不是每个内部片段都显示完成态。

#### 产品口径

- 完成态应聚合表达，避免每个低层事件都生成独立可见节点。
- 同一组连续成功事件可以只保留一个完成状态，或直接通过该组标题 / 子项状态表达。
- 成功完成不应比失败、等待用户、正在运行更抢眼。
- 文件产物项本身已经表达“已生成 / 已写入”时，不需要紧跟一个额外的完成勾节点。

#### 建议规则

| 场景 | 展示建议 |
|------|----------|
| 连续多个内部工具成功结束 | 合并为一个完成状态或隐藏子级完成勾 |
| 文件写入成功 | 优先展示文件项；不额外展示空白完成勾 |
| 思考过程折叠项结束 | 折叠项自身体现完成，不追加独立完成节点 |
| 失败 / 中断 / 等待用户 | 仍保留明确状态图标和文案 |
| 当前正在运行 | 保留运行态图标，避免用户不知道系统是否还在处理 |

#### 验收标准

- 同一轮回复中不出现多个连续、无文案或低信息量的完成勾图标。
- 文件产物附近不再出现重复完成勾，用户能清楚看出哪些是文件、哪些是执行状态。
- 时间线仍能表达“运行中 / 已失败 / 等待用户 / 已完成”这些关键状态。
- 视觉重心回到步骤标题、工具摘要、文件产物和最终正文，而不是结束图标本身。

#### 实现备注

`0.1.1-alpha` 已先完成可见层收敛：时间线仅在有明确步骤标题的 episode 上展示完成勾；无标题文件 / 产物 / 说明步骤不再额外显示空白完成勾；思考过程内部不再追加独立“结束”完成行，避免与外层时间线完成态重复。

### A011-CHAT-002 对话记录更多菜单

#### 背景

左侧栏对话记录目前主要用于进入历史会话，但缺少对单条记录的轻量管理能力。用户在历史列表中看到某条对话时，应该可以直接修改标题、打开该对话对应的本机工作文件夹，或从列表中移除该记录。

#### 触发位置

- 左侧栏项目分组下的单条对话记录。
- 鼠标悬浮到对话记录行时，在时间位置或行尾显示“更多”按钮。
- 点击“更多”按钮打开菜单。

#### 菜单项

| 菜单项 | 行为 | 说明 |
|--------|------|------|
| 重命名 | 修改该对话记录标题 | 只改历史记录标题，不改本地文件夹名，除非后续明确增加“同步重命名文件夹”能力 |
| 打开文件夹 | 在系统文件管理器中打开该对话绑定的本机工作文件夹 | 对系统默认创建项目，打开任务文件夹；对用户手动项目内的会话，打开对应项目文件夹或会话产物目录，具体以当前数据模型可定位范围为准 |
| 移除 | 从左侧历史列表移除该对话记录 | 默认只移除会话记录，不删除本机文件夹和文件 |

#### 交互规则

- 默认状态下保持列表简洁，只显示标题、状态点和时间。
- 悬浮到记录行后，时间可被“更多”按钮替换，或在时间旁出现更多按钮。
- 更多按钮点击区域不能影响点击整行进入会话。
- 菜单打开时，应保持当前行高亮。
- 点击菜单外、按 `Esc`、切换路由或滚动导致行消失时关闭菜单。
- 当前正在运行的会话可以允许重命名和打开文件夹；移除若存在中断风险，应禁用或要求二次确认。

#### 数据与安全边界

- “重命名”只更新会话标题 / 侧栏展示标题。
- “打开文件夹”必须走本机安全路径解析，不允许打开项目根之外的任意路径。
- “移除”不得删除本地磁盘文件。
- “移除”不得影响同一项目下其他会话。
- 若移除当前打开的会话，应跳转到同项目的新对话页、最近会话或安全默认页。

#### 异常状态

| 场景 | 期望表现 |
|------|----------|
| 对话没有可定位文件夹 | “打开文件夹”置灰或提示暂无可打开文件夹 |
| 本机文件夹不存在 | 显示轻量错误提示，并保留对话记录 |
| 重命名为空 | 禁止保存，保留原标题 |
| 移除失败 | 保留对话记录，显示轻量错误提示 |

#### 验收标准

- 鼠标悬浮到左侧对话记录时，可以看到“更多”按钮。
- 点击“更多”后出现“重命名 / 打开文件夹 / 移除”三个操作。
- 重命名后，左侧记录标题立即更新，刷新后仍保留。
- 打开文件夹能定位到该对话对应的本机工作文件夹或项目文件夹。
- 移除后，该对话从左侧历史列表消失，且不会删除本机文件。

#### 实现备注

`0.1.1-alpha` 已接入对话记录悬浮更多按钮：支持本地标题重命名、通过 Desktop IPC 打开会话所属项目文件夹、从本地历史索引移除记录。移除仅影响侧栏历史索引，不删除本机文件。

### A011-CHAT-003 任务状态条展示任务名与状态

#### 背景

当前输入区上方的任务状态条只显示类似“当前任务已完成 · 3/3”的信息。这个表达只能告诉用户数量进度，不能说明当前任务是什么、执行到了哪一步、哪些任务已经完成、是否还有任务正在执行。

#### 问题描述

- `3/3` 缺少上下文，用户不知道 3 个任务分别是什么。
- “当前任务已完成”没有展示任务名，无法确认完成的是哪个任务。
- 执行中、已完成、失败、等待用户等状态没有被清晰区分。
- 当一轮回复包含多个工具调用 / 文件产物 / 子任务时，状态条无法帮助用户建立执行预期。

#### 需求说明

- 任务状态条应展示当前任务名称或当前步骤名称。
- 任务状态条应展示明确状态，例如：执行中、已完成、失败、等待确认、已取消。
- 数字进度 `3/3` 可以保留，但应作为辅助信息，而不是唯一主体。
- 如果有多个任务，应能让用户理解当前正在处理哪一个，以及整体进度。

#### 展示建议

| 状态 | 示例文案 |
|------|----------|
| 执行中 | `正在执行：生成参数化图纸 · 2/3` |
| 已完成 | `已完成：生成参数化图纸 · 3/3` |
| 失败 | `失败：导出 STL 文件 · 2/3` |
| 等待用户 | `等待确认：选择导出格式 · 1/3` |
| 已取消 | `已取消：生成 PPT 初稿 · 1/3` |

#### 交互规则

- 状态条默认展示当前最重要的任务 / 步骤。
- 任务名过长时单行截断，但应保留状态词和进度数字。
- 鼠标悬浮状态条时，可展示完整任务名或任务摘要。
- 点击状态条可以展开查看任务列表或跳转到当前执行位置；若本期不实现展开，也应为后续保留交互位。
- 关闭状态条后，不应影响任务继续执行。

#### 验收标准

- 状态条不再只显示“当前任务已完成 · 3/3”这类低上下文文案。
- 用户能从状态条看出当前任务名 / 步骤名。
- 用户能区分执行中、完成、失败、等待用户、取消等关键状态。
- `n/n` 进度数字仍可见，但不是唯一信息。
- 任务名过长时不撑破输入区布局。

#### 实现备注

`0.1.1-alpha` 已优化 `PinnedTodoBar` 展示：状态条优先显示“进行中 > 待执行 > 最近完成”的任务名，同时展示执行中 / 完成 / 待执行 / 已取消状态与 `n/n` 进度数字。

### A011-AGENT-001 记住智能体与模型选择偏好

#### 背景

当前智能体 / 模型选择面板每次打开或进入新会话时，容易回到 `Codex` / `Codex 默认`。如果用户上次已经选择了其他 CLI 智能体、模型或模型 API，系统仍然回退到 Codex，会让用户感觉偏好没有被尊重，也容易导致下一轮任务使用了错误的执行环境。

#### 需求说明

- 智能体选择器应默认读取用户上次选择的智能体偏好。
- 模型选择器应默认读取该智能体下用户上次选择的模型偏好。
- 用户切换智能体或模型后，应立即保存偏好。
- 新建会话、切换模块、刷新页面、重启应用后，应优先恢复上次偏好。
- 只有当上次偏好不可用时，才允许回退到系统默认项。

#### 回退规则

| 场景 | 期望表现 |
|------|----------|
| 上次选择的智能体仍可用 | 默认选中该智能体 |
| 上次选择的模型仍可用 | 默认选中该模型 |
| 上次智能体不可用 / 未安装 / 未登录 | 回退到可用智能体，并提示或弱提示当前已回退 |
| 上次模型不存在或不可用 | 回退到该智能体的默认模型 |
| 用户首次使用 | 使用系统默认智能体与默认模型 |

#### 验收标准

- 用户选择非 Codex 智能体后，新建会话不再自动回到 Codex。
- 用户选择某个模型后，刷新页面或重启应用仍能恢复该模型。
- 不同模块入口中的选择器遵循同一套偏好，不出现对话页记住、写作 / PPT 页不记住的割裂。
- 当偏好不可用时，系统能安全回退到可用默认项，不阻塞用户继续发送任务。

#### 实现备注

`0.1.1-alpha` 已补齐 CLI 智能体选择持久化：用户选择某个 CLI 智能体 / 模型时，会同时写入 `executionSource`、`defaultAgentId` 与该智能体对应的 `agentModels` 偏好；模型 API 选择继续复用 `activeApiSelection`。新建会话、模块切换和刷新时优先从设置恢复上次偏好，不可用时走现有可用项回退。

### A011-WORKSPACE-001 工作区新增文件 / 文件夹能力

#### 背景

当前工作区文件树顶部已经展示“添加文件”和“添加文件夹”两个按钮，但实际逻辑仍是前端占位，没有真正写入本机工作区。用户点击后应能在当前项目目录中创建真实文件或文件夹，并立即在文件树中看到结果。

#### 目标

- “添加文件”真实创建当前项目内的文件。
- “添加文件夹”真实创建当前项目内的文件夹。
- 创建结果作用于当前绑定项目或系统默认创建项目的本机目录。
- 创建后文件树刷新，用户能继续编辑或使用新建内容。

#### 交互规则

| 操作 | 交互 | 成功后 |
|------|------|--------|
| 添加文件 | 显示内联文件名输入行，默认 `untitled.md` | 创建文件，刷新文件树，自动打开该文件 |
| 添加文件夹 | 显示内联文件夹名输入行，默认 `new-folder` | 创建文件夹，刷新文件树，展开或定位该文件夹 |

#### 目标位置规则

- 如果当前选中的是文件夹：在该文件夹下创建。
- 如果当前选中的是文件：在该文件所在目录创建。
- 如果没有选中项：在当前项目根目录创建。
- 支持输入相对路径，例如 `notes/brief.md`、`exports/images`。

#### 接口建议

新增 Web BFF：

```http
POST /api/workspace/entry
```

创建文件：

```json
{
  "projectId": "proj-xxx",
  "type": "file",
  "path": "notes/brief.md",
  "content": ""
}
```

创建文件夹：

```json
{
  "projectId": "proj-xxx",
  "type": "folder",
  "path": "exports/images"
}
```

Companion 可采用以下任一实现：

- 新增 `POST /v1/projects/:projectId/entries`，统一创建文件 / 文件夹。
- 或复用现有 `PUT /v1/projects/:projectId/files` 创建文件，新增 `POST /v1/projects/:projectId/folders` 创建文件夹。

#### 安全规则

- 路径必须是项目内相对路径。
- 禁止绝对路径。
- 禁止 `../` 或任何逃逸项目根目录的路径。
- 禁止空名称、只包含空格的名称。
- 默认不覆盖已有文件或文件夹。
- 当前 `projectId = none` 或工作区不可访问时，不允许静默失败，应提示用户先创建 / 选择工作文件夹或重试。

#### 前端改造点

| 文件 | 改造 |
|------|------|
| `web/src/components/workspace/FileTree.tsx` | 将“添加文件 / 添加文件夹”按钮从占位日志改为真实创建动作 |
| `web/src/components/workspace/WorkspaceContext.tsx` | 增加 `createWorkspaceFile` / `createWorkspaceFolder`，封装刷新与打开逻辑 |
| `web/src/lib/workspace/adapter.ts` | 增加 `createWorkspaceEntry` 请求封装 |
| `web/src/app/api/workspace/entry/route.ts` | 新增 BFF 路由，解析 `projectId` 并转发 Companion |

#### Companion 改造点

| 文件 | 改造 |
|------|------|
| `companion/src/projects/tree.ts` | 增加创建文件夹能力；创建文件时补充存在性 / 目录逃逸检查 |
| `companion/src/routes/projects.ts` | 暴露创建 entry / folder API |

#### 异常状态

| 场景 | 期望表现 |
|------|----------|
| 文件已存在 | 提示已存在，不覆盖 |
| 文件夹已存在 | 提示已存在，可视为成功或提示已存在，口径需统一 |
| 路径非法 | 提示路径不可用 |
| 工作区不可访问 | 提示“工作文件夹不存在或不可访问”，保留重试入口 |
| 创建失败 | 不刷新为假成功，展示轻量错误提示 |

#### 验收标准

- 点击“添加文件”后，本机项目目录中出现真实文件。
- 点击“添加文件夹”后，本机项目目录中出现真实文件夹。
- 创建后文件树自动刷新。
- 新建文件自动打开，用户可继续编辑保存。
- 不允许创建到项目目录外部。
- 文件 / 文件夹已存在时不会被静默覆盖。

#### 实现备注

`0.1.1-alpha` 已完成基础闭环：工作区文件树按钮已接入真实创建动作；Web 新增 `/api/workspace/entry` BFF；Companion 新增 `/v1/projects/:projectId/entries` 与 `createProjectEntry`，支持创建文件 / 文件夹、禁止绝对路径与目录逃逸、默认拒绝覆盖已有 entry。前端采用文件树内联新建输入行，创建文件后会刷新文件树并打开新文件；创建文件夹后会刷新文件树并展开新文件夹。
