# 平台与工作区边界定义

## 目标

明确 Web 与 Desktop 两个平台的产品边界、工作区来源、数据归属和交互原则，避免在后续设计与实现中混淆两端的运行方式与存储语义。

## 结论

- 当前平台发布版本是 `0.1.0-alpha`，产品阶段是 Desktop Alpha。
- Web 是在线沙箱产品。
- Desktop 是本地文件夹产品。
- 当前 Desktop 导航范围为六个模块：对话、写作、PPT、3D绘图、视频、推演。
- 对话、写作、PPT 是 Desktop Alpha / Beta 主验收线；3D绘图、视频、推演作为 0.x / M1 / Beta 子线并行推进。
- 两端共享能力抽象，不默认共享工作区数据。

## 平台定义

| 平台 | 产品定位 | 工作区类型 | 数据归属 | 默认目标 |
|---|---|---|---|---|
| Web | 在线工作台 | `sandbox workspace` | Web 侧沙箱存储 | 打开即用 |
| Desktop | 本地工作台 | `local folder workspace` | 用户本机文件夹 | 处理真实本地项目 |

## 工作区模型

### Web

Web 仅提供 `sandbox workspace`。

- 用户打开 Web 后，直接进入在线工作区。
- 文件、上传资料、生成文档、临时中间结果都保存在沙箱工作区。
- Web 不提供本地目录选择、本地路径录入或本机运行时连接入口。
- Web 不是本地运行模式，不能依赖用户本机 Companion 才可使用。

### Web 云端服务要求

如果 Web 被定义为真正可用的在线沙箱产品，则必须具备云端服务能力。

至少需要以下能力：

- Web API / 应用服务
- Sandbox workspace 管理
- 云端文件存储
- 云端任务运行
- 运行状态与产物回传

这意味着 Web 端不应仅仅是前端页面，而应建立完整的在线工作区能力闭环。

Web 的基本分层建议如下：

- Web Frontend：聊天、文件树、预览、上传、运行状态展示
- Web API / App Server：账号、会话、工作区元数据、上传授权、任务编排
- Sandbox Runtime Service：沙箱工作区、文件存储、任务运行、产物生成

### Desktop

Desktop 仅提供 `local folder workspace`。

- 工作区就是用户显式选择的本地文件夹。
- 文件读写、文件树、终端、CLI 运行都围绕该本地文件夹展开。
- Desktop 允许在用户未显式选择项目时，自动创建一个默认本地工作区。
- 该默认工作区必须是真实的本地文件夹，而不是内部托管沙箱。
- 后续启动默认恢复上次打开的本地工作区。

### Desktop 默认工作区策略

Desktop 的正式要求如下：

- 桌面端没有问题时，不要求用户每次都先手动选择项目。
- 当用户尚未选择项目时，系统可以自动新建一个默认工作区。
- 这个默认工作区的语义仍然是 `local folder workspace`。
- 该目录应落在用户本机可见、可访问的真实路径下。
- 用户后续可以继续使用这个默认本地工作区，也可以切换到其他本地文件夹。

这意味着桌面端的“默认新建项目”不是例外逻辑，也不是待修问题，而是产品定义的一部分。

## 当前模块范围

当前版本中，Desktop 导航可见以下六个业务模块：

- 对话
- 写作
- PPT
- 3D绘图
- 视频
- 推演

以下模块不进入当前 Desktop Alpha / Beta 产品路线：

- 翻译
- 会议纪要
- 知识库
- 其他非上述六模块的扩展业务入口

其中，会议纪要作为后续版本候选能力保留评估空间；知识库明确不需要，不再建设独立模块。

## 数据边界

默认规则如下：

- Web 数据不自动出现在 Desktop。
- Desktop 数据不自动出现在 Web。
- 两端工作区不是同一个存储域，不应被建模为同一个 project 或同一份文件系统。

这意味着两端的数据默认不互通。

## 为什么默认不互通

Web 与 Desktop 的底层语义不同：

- Web 使用沙箱存储。
- Desktop 使用用户本机真实目录。
- 两端的权限模型、生命周期、路径语义和存储位置都不同。

如果强行做自动互通，会立即引入以下问题：

- 同步策略复杂
- 文件冲突处理复杂
- 权限与隐私边界不清
- 用户误解两端是否在操作同一份文件

因此默认不互通是更合理的产品选择。

## 允许的跨端流转方式

虽然默认不互通，但可以允许显式的数据流转：

- 下载
- 上传
- 导出
- 导入
- 交付物分享

这些能力应被定义为显式动作，而不是隐式同步。

## 共享与不共享

### 可以共享的内容

- 聊天与任务流
- 工作区概念抽象
- 文件树与预览的 UI 结构
- 运行状态协议
- 产物展示协议
- 业务模块能力
- 部分前端组件与页面框架

### 不共享的内容

- 工作区来源
- 存储位置
- 权限模型
- 运行时接入方式
- 工作区生命周期
- 底层运行环境

## 运行时边界

Web 与 Desktop 应采用不同的底层运行时。

### Web 运行时

Web 应连接云端 sandbox runtime。

- 面向在线工作区
- 负责 Web 的文件、运行、产物与执行环境
- 不依赖用户本机环境

### Desktop 运行时

Desktop 应连接本机 Companion runtime。

- 面向本地文件夹
- 负责本地文件树、本地终端、本地 CLI 与本地依赖环境
- 直接运行在用户设备上

## 推荐架构结论

两端推荐采用以下结构：

- Web -> 云端 Sandbox Service -> 在线工作区
- Desktop -> 本机 Companion -> 本地文件夹

两端共享上层产品能力与协议抽象，但底层运行时分离。

## 版本策略

当前版本与后续版本的实施策略如下：

- `0.1.0-alpha` / Desktop Alpha：Desktop 继续沿用现有本地工作区方案，完成对话主链路与桌面壳基线。
- `0.2.0-beta` / Desktop Beta：收口对话、写作、PPT 主线真实链路，并完善 HMAC、托盘、Companion 捆绑、自动更新等桌面能力；3D绘图、视频、推演按子线节奏验收。
- Web Sandbox 1.0 / 下一大版本：建设完整云端 sandbox runtime、在线工作区、云端文件存储、多用户后台与协作能力。
- Desktop 不做架构性减法，继续按既有本地工作区逻辑演进六个可见模块。
- Web 端升级为完整云端 sandbox 产品的工作，放到下一个大版本推进。

这意味着当前阶段的明确结论是：

- Desktop 不需要因为 Web 的目标架构而做减法。
- Web 不要求在当前版本内完成向云端 sandbox 架构的彻底升级。
- Web 的云端服务化、sandbox runtime、在线工作区闭环等能力，作为下一大版本的正式演进方向。

因此，本文件中的 Web sandbox 目标是后续版本边界，不作为当前版本必须完成的改造项。

## 交互原则

### Web 交互原则

- 打开即进入在线工作区。
- 首页重点入口是继续工作、上传资料、开始任务。
- 不向用户暴露本地路径、本地目录或本机环境概念。
- 对用户可以统一称为“工作区”或“在线工作区”。

### Desktop 交互原则

- 首次启动时，如果尚无工作区，可以自动创建默认本地工作区，或引导用户选择本地文件夹。
- 后续启动时自动恢复上次工作文件夹。
- 明确展示当前本地工作区位置。
- 强化围绕本地文件夹的操作体验，例如文件树、终端、CLI 和本地预览。

## 对外文案建议

- Web：在线工作区
- Desktop：本地工作区

如果产品层面统一使用“工作区”一词，则应在辅助文案中补充“在线”或“本地”，避免用户误解两端默认互通。

## 不建议的方案

以下方案不建议采用：

- Web 与 Desktop 默认共享同一份工作区数据
- 将 Web 沙箱与 Desktop 本地文件夹映射为同一个底层项目
- 在 Web 中引入本地路径或本地工作区心智
- 在 Desktop 中增加与需求无关的沙箱工作区模式
- 让 Web 默认依赖本机 Companion 才能工作

## 最终产品定义

- Web：在线沙箱工作区，打开即用
- Desktop：本地文件夹工作区，面向真实本地项目
- 两端共享能力抽象，不默认共享工作区数据
