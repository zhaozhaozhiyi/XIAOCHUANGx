| 属性 | 内容 |
|------|------|
| 文档版本 | v1.0 |

| 状态 | **待评审（P0/P1）** |

| 适用范围 | Web 对话页、桌面壳对话页、Companion/Agent 运行过程展示 |
| 核心目标 | 将当前“事件流直出”优化为“结论优先、阶段可感知、证据可追溯、技术细节默认隐藏”的对话体验 |

---

## 0. 背景

当前对话模块已经具备较完整的分块与时间线能力：

- 有 `turn_meta`、`tool_batch`、`narration`、`reasoning`、`deliverables` 等 `parts[]` 基础能力；
- 支持按 `streamSeq` 进行时间序交错渲染；
- 支持工具过程聚合、Pinned Todo、文件读写与交付物展示。

但从用户感知看，当前实现仍偏“研发视角”：

1. **过程与结果混排过深**：`summary / narration / tool / reasoning / status` 同层竞争注意力。
2. **过程更像事件日志**：展示的是“发生了什么工具动作”，而不是“任务推进到了什么阶段”。
3. **技术信息默认暴露过多**：`reasoning`、`skill`、单条 `tool`、`status_chip` 对业务用户理解成本高。
4. **完成态收束不足**：用户读完整条消息时，容易感受到“过程很多、每轮都差不多、重复感强”。

本规格用于定义下一阶段的对话模块优化方向，重点解决“过程展示该不该看、看什么、默认怎么收”的问题。

---

## 1. 设计目标

### 1.1 目标

1. **结果优先**：完成态优先展示结论、建议、交付物，而不是内部过程。
2. **阶段可感知**：进行中让用户知道“现在在做什么”，而不是只看到耗时或工具名。
3. **证据可追溯**：保留工具过程作为证据层，支持高级用户与研发排查。
4. **噪音可收束**：默认隐藏推理、skill、原始状态等内部细节，减少重复感。
5. **兼容现有协议**：优先复用当前 `parts[]` 与 reducer，不强依赖后端协议重构。

### 1.2 非目标

1. 本次不重新设计整个聊天页面布局。
2. 本次不引入新的多栏信息架构。
3. 本次不要求 Companion/daemon 一次性新增完整结构化阶段协议。
4. 本次不默认向普通用户暴露完整推理链。

---

## 2. 用户与场景

### 2.1 用户分层

| 用户类型 | 关注点 | 默认应看到什么 |
|----------|--------|----------------|
| 业务用户 | 结论、进度、是否需要自己介入 | 结论、当前阶段、阻塞提示 |
| 高级用户 | 依据、过程、是否真实执行 | 结论 + 可展开过程证据 |
| 研发/测试 | 工具链路、状态流、失败点 | 结论 + 过程证据 + 技术详情 |

### 2.2 典型场景

1. 资料检索 / 代码分析：用户需要知道系统真的读了哪些信息源，但不想逐条看工具日志。
2. 长时任务：用户需要知道目前是在分析、在等待还是已卡住。
3. 半自动场景：如“等待选择目录”“等待权限确认”，用户必须明确知道该自己做什么。
4. 交付场景：用户更关心结论和文件成果，过程只作为辅助证明。

---

## 3. 体验原则

### 3.1 总原则

1. **结论是主角，过程是配角。**
2. **阶段说明优先于事件清单。**
3. **用户语言优先于系统术语。**
4. **默认展示业务可理解信息，技术细节进入折叠层。**
5. **完成态自动收束，避免长串重复过程平铺。**

### 3.2 应避免的问题

1. 默认展示 `connect`、`accepted`、`requesting`、skill slug、内部 run id 等研发态字段。
2. 多条过程连续出现但没有信息增量，如反复“我会继续检查”“我将继续定位”。
3. `reasoning` 默认展开。
4. 工具批次完成后仍长期占据主视图。
5. 让用户从过程区反推“现在到底有没有结果”。

---

## 4. 目标信息架构

单条 assistant 消息的展示层级调整为：

```text
1. 状态条（整轮状态 / 阶段）
2. 当前阶段说明（进行中时）
3. 最终结论正文
4. 交付物 / Todo / 可执行结果
5. 处理过程（可折叠）
6. 技术详情（默认隐藏）
```

### 4.1 三层视图定义

| 层级 | 目的 | 默认状态 |
|------|------|----------|
| 主视图 | 给普通用户快速理解任务结果与当前状态 | 默认可见 |
| 过程视图 | 提供过程证据，证明真实执行过 | 默认折叠 |
| 调试视图 | 提供研发排障所需底层信息 | 默认隐藏 |

---

## 5. `part` 分层落位规则

基于当前 `parts[]`，展示策略调整如下。

### 5.1 主视图

默认进入主视图的 `part`：

| part kind | 作用 | 展示策略 |
|-----------|------|----------|
| `turn_meta` | 整轮状态/耗时/阶段 | 始终可见，位于最上方 |
| `narration` | 阶段说明 | 进行中时保留最近 1 条 |
| `summary` / `text` | 结论正文 | 始终可见 |
| `error` | 异常结果 | 始终可见 |
| `todo` | 需跟进事项 | 保持现有 Pinned 方案 |
| `artifact` / `deliverables` | 交付成果 | 紧随正文展示 |

### 5.2 过程视图

默认收进“查看处理过程”的 `part`：

| part kind | 作用 | 展示策略 |
|-----------|------|----------|
| `tool_batch` | 聚合后的过程证据 | 完成态默认折叠 |
| `file_read` / `document_read` | 读取证据 | 归入过程视图 |
| `file_edit` / `document_edit` | 修改证据 | 归入过程视图 |
| `command` | 执行命令证据 | 归入过程视图 |
| 单条 `tool` | 原始工具调用 | 仅过程或调试视图 |

### 5.3 调试视图

默认进入“查看技术详情”的 `part`：

| part kind | 原因 |
|-----------|------|
| `reasoning` | 成本高、噪音大、非普通用户信息 |
| `skill` | 研发概念，非业务概念 |
| `status_chip` | 偏内部状态标签 |
| 原始 `status` | 若语义偏内部态，则仅技术详情展示 |

备注：

- 若某条 `status` 已翻译成用户语言，如“等待你选择目录”，则可升级进入主视图。
- `reasoning` 在调试视图中也应默认折叠。

## 6. 状态设计

### 6.1 标准状态

| runStatus | 用户态名称 | 主展示 |
|-----------|------------|--------|
| `running` | 进行中 | 当前阶段 + 可选最近过程 |
| `waiting_user` | 等待用户 | 明确阻塞原因与动作 |
| `complete` | 已完成 | 结论 + 成果，过程折叠 |
| `error` | 失败 | 错误说明 + 已完成部分 |
| `cancelled` | 已中断 | 中断提示 + 保留已有结果 |

### 6.2 状态条文案规范

状态条不应只显示耗时，应优先表达任务阶段。

推荐文案：

- `正在分析库存变化`
- `正在检索参考数据`
- `正在整理结论`
- `等待你选择目录`
- `已完成，用时 43 秒`
- `处理失败，可重试`

不推荐文案：

- `已处理 43846ms`
- `running`
- `accepted`
- `requesting`

### 6.3 耗时信息规则

1. 耗时作为辅助信息保留。
2. 进行中时可展示“已处理 43s”，但不应压过阶段名。
3. 完成态可写成“已完成，用时 43 秒”。
4. 毫秒级耗时不应面向普通用户显示。

---

## 7. 进行中与完成态行为

### 7.1 进行中

进行中消息应以“阶段感知”为核心：

1. 显示状态条。
2. 显示最近一条 `narration`。
3. 若尚无正文，只显示最近过程摘要，不平铺完整事件流。
4. `tool_batch` 只展开当前批次。
5. `reasoning`、`skill` 等不进入默认主视图。

### 7.2 完成态

完成后应自动收束为：

1. 结论正文保留在前。
2. 交付物紧跟正文。
3. 过程区折叠为一句摘要 + 展开入口。
4. 技术详情继续默认隐藏。

推荐结构：

```text
已完成，用时 43 秒
结论：……
附件：……
[查看处理过程]
[查看技术详情]
```

## 8. 过程视图设计

### 8.1 过程视图目标

过程视图回答的是：

1. 这轮任务是否真实执行过？
2. 系统大致做了哪些动作？
3. 结论是否有迹可循？

### 8.2 过程摘要规则

过程区首行应优先展示聚合摘要，而不是原始工具枚举。

示例：

- `已读取 3 个文件，搜索 2 次，运行 1 条命令`
- `已检索 2 个来源，确认 1 条链路`
- `已探索 5 个文件`

展开后再看明细：

- 读取了哪些文件
- 执行了哪些命令
- 是否成功

### 8.3 `tool_batch` 要求

1. 进行中默认展开。
2. 完成后默认折叠。
3. 标题必须是人话摘要，不能直接堆叠原始 tool 名。
4. 同一轮中多个低价值 batch 应允许合并或弱化。

---

## 9. 调试视图设计

### 9.1 适用对象

调试视图主要供：

- 研发
- 测试
- 高级用户

### 9.2 内容范围

可包含：

- 推理过程
- skill 注入信息
- 原始 status / status chip
- 单条 tool 粒度信息
- 原始命令与读写细节

### 9.3 展示要求

1. 默认折叠，不进入主时间线视觉中心。
2. 与“处理过程”分开，避免普通用户误把调试信息当结果。
3. 允许后续接设置项，如“默认展开技术详情（开发模式）”。

---

## 10. 文案规范

### 10.1 阶段说明文案

每条阶段文案应回答三件事：

1. 我刚刚确认了什么
2. 这意味着什么
3. 我下一步要做什么

示例：

- `我已经确认库存数据入口，下一步会核对上周环比口径。`
- `我已定位到桌面导入链路，接下来验证真实联调是否打通。`

### 10.2 禁止文案

1. 无增量重复表述，如连续多条“我会继续检查”。
2. 未解释的工程术语直接裸露给业务用户。
3. 单纯复述工具名而没有任务语义。

## 12. 分阶段实施建议

### P0：纯前端展示层优化

1. 调整 assistant 消息层级，改为结果优先。
2. 完成态默认折叠 `tool_batch`。
3. `reasoning` 默认不进主视图。
4. 增加 `waiting_user` 的明确展示。

### P1：轻量协议增强

1. 增加更稳定的阶段文案来源，如 `stageLabel`。
2. 增加面向展示层的 `audience` 或 `debugOnly` 标记。
3. 支持过程区更稳定的聚合摘要。

### P2：高级模式

1. 增加“开发者模式 / 调试视图”开关。
2. 支持更细粒度的技术详情展开。
3. 支持按用户类型记忆展示偏好。

---

## 13. 验收标准

### 13.1 用户体验

- [ ] 完成态消息中，用户首先看到的是结论而不是过程列表。
- [ ] 普通用户不再默认看到 `reasoning`、skill slug、内部状态词。
- [ ] 进行中能明确感知当前阶段，而不是只看到耗时。
- [ ] “等待用户”场景会明确提示下一步动作。

### 13.2 过程可信度

- [ ] 过程区仍可追溯到文件读取、检索、命令执行等证据。
- [ ] 过程摘要是用户语言，不是原始工具名拼接。
- [ ] 完成后过程区默认折叠，但可一键展开查看。

### 13.3 技术实现

- [ ] 不破坏现有 `parts[]` 协议兼容性。
- [ ] 不破坏 `streamSeq` 时序能力。
- [ ] 可在不改后端的前提下完成 P0 方案。

---

## 14. 一句话结论

对话模块的优化方向不是“展示更多过程”，而是把现有过程能力重新分层：**让结果更靠前、阶段更清晰、证据可追溯、技术细节退居二线。**
